# Y-Link 项目全局规范

## 0. Codex 工作总则

- 本文件是 Y-Link 仓库级开发规范，所有代码修改、重构、修复、测试、文档更新均必须遵守。
- 默认使用简体中文回答；所有代码注释、UI 文本、计划书、提交说明和回复总结也必须使用简体中文。
- 修改代码前必须先阅读项目结构和相关文件，理解真实业务链路后再动手。
- Codex 在动手修改前，必须先判断当前任务是否命中仓库内 `.agents/skills/*/SKILL.md` 中定义的 skill。
- 若任务命中 skill，必须优先使用对应 skill 的工作流；若未使用，应在最终回复中说明未使用原因。
- 不要无理由重构，不要扩大改动范围，不要为了快速修复绕过既有服务层、权限层、路由层、审计层或验证脚本。
- 不要引入新依赖，除非先说明原因、收益和风险。
- 不要泄露密钥、Token、Cookie、数据库密码、永久删除密码、Webhook 完整地址等敏感信息。
- 修改后必须说明改了哪些文件、为什么改、如何测试以及测试结果。
- 遇到风险操作，例如删除文件、数据库迁移、大规模重构、权限收紧、数据清理等，必须先说明风险。

## 1. 强制性能验证

- **触发时机**：每次大版本升级、核心页面重构、路由结构调整、依赖大版本更新、核心业务链路重构后，必须执行性能与回归验证。
- **验证方式**：在提交或结束当前大迭代前，必须运行：

```bash
npm run verify:performance
```

- 如果验证因本地环境、沙箱权限或外部依赖失败，必须记录失败原因、失败阶段和后续处理建议。

---

## 2. Vue 页面/组件头部注释规范

- 新增或重构 Vue 页面、工作台、弹窗、抽屉等核心组件时，必须在文件头部补充中文说明注释。
- 头部注释必须优先采用以下结构，避免空泛描述：

```ts
/**
 * 模块说明：src/views/inbound/SupplierHistoryView.vue
 * 文件职责：展示供货方历史送货单列表，并支持服务端筛选分页、详情查看和待入库二维码回查。
 * 实现逻辑：
 * - 页面采用“统计卡 + 筛选工具栏 + 列表容器 + 详情抽屉”的工作台布局，与录入页形成统一视觉语言；
 * - 历史页改为服务端筛选与分页，只返回当前页数据，降低首屏等待与前端内存占用；
 * - 列表请求与详情请求都接入稳定请求工具，避免快速切换筛选或连点详情时旧结果覆盖新状态；
 * - 详情抽屉保留原有查询与二维码逻辑，仅增强打开反馈与信息层级，不改动任何业务接口。
 * 维护说明：
 * - 若后续要增加更多筛选维度，优先继续扩展服务端查询参数，而不是回退到前端全量筛选；
 * - 二维码生成失败时不能静默吞掉，否则用户只会看到空白占位而不知道单据本身已存在。
 */
```

- 注释内容必须贴合当前文件的真实职责，不能机械复制示例路径或业务内容。
- 不要为了满足注释规范添加空泛注释；注释应说明真实职责、关键实现逻辑和后续维护边界。

## 3. Element Plus 使用规范

- 管理端和客户端可交互控件优先使用 Element Plus 原生组件，例如 `el-button`、`el-input`、`el-select`、`el-checkbox`、`el-radio-group`、`el-upload`、`el-table`、`el-pagination`、`el-menu`。
- 新增交互控件时不要回退到原生 `button`、`input`、`textarea`、`select`、`dialog`，除非是打印/PDF 模板等明确需要原生结构保持纸张版式的场景。
- 使用 Element Plus 时必须遵守当前 API，避免 `label act as value`、`small` 等即将废弃写法。
- 替换组件时保持现有业务逻辑和 UI 样式基本不变，不得借机重设计页面。
- 高频刷新、长列表、自动同步区域只允许轻量动画，必须考虑性能和 `prefers-reduced-motion` 降级。

---

## 4. 安全与权限规范

- 不允许在 URL query 中传递访问 token；管理端和客户端鉴权应使用安全 Cookie 或 Bearer Header。
- 通知外发 URL、Webhook、第三方地址一律视为不可信输入，必须做协议、域名和内网地址校验。
- 日志、审计、错误提示和前端展示中不得输出完整 token、密码、Cookie、Webhook 密钥路径或其他敏感值。
- 匿名查询、高风险维护接口、永久删除、数据库迁移等能力必须有明确频控、权限或管理员角色门禁。
- 永久删除类操作必须按服务端配置的永久删除密码校验，不允许前端本地比对，不允许记录明文密码。

---

## 5. 验证与交付规范

- 修改核心业务代码后，必须运行对应验证命令；无法运行时必须说明原因。
- 前端核心改动至少验证：

```bash
npm run build
```

- 涉及编码、中文文本或文件批量改动时，必须验证：

```bash
npm run verify:text-encoding
```

- 涉及权限、路由、鉴权、安全边界时，应补充对应后端验证脚本，例如：

```bash
npm --prefix backend run task2:route-contract:verify
npm --prefix backend run security:hardening:verify
```

- 涉及后端 TypeScript 代码时，必须验证：

```bash
npm --prefix backend run build
```

- 最终回复必须明确列出实际运行过的命令和结果；不要声称运行了未实际运行的测试。
