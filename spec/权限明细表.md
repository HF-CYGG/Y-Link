# 权限明细表（管理员 / 操作员）

## 文件说明
- 本表为独立权限明细清单，面向上线核对、角色开通与权限审计。
- 口径来源：`backend/src/constants/auth-permissions.ts` 与当前已落地权限隔离策略。
- 角色定义：
  - 管理员（admin）：可管理系统与关键数据，承担治理责任。
  - 操作员（operator）：仅处理日常业务动作（售货、开单、核销、入库核验等）。

## 1. 角色权限总览

| 角色 | 权限数量 | 核心能力 |
|---|---:|---|
| 管理员（admin） | 20 | 全量业务操作 + 系统治理 + 用户治理 + 审计 |
| 操作员（operator） | 11 | 日常业务操作（不开启关键治理权限） |

## 3. 管理员相对操作员新增权限（重点隔离）

| 权限码 | 管理风险级别 | 隔离说明 |
|---|---|---|
| `orders:delete` | 高 | 影响核心业务单据完整性，操作员禁止 |
| `system_configs:update` | 高 | 影响系统行为与全局参数，操作员禁止 |
| `users:view` `users:create` `users:update` `users:status` `users:reset_password` | 高 | 用户治理仅管理员可执行 |
| `audit_logs:view` `audit_logs:export` | 高 | 审计数据仅管理员可访问 |

## 5. 上线核对建议
- 使用 `admin` 与 `operator` 各登录一次，分别验证：菜单可见性、按钮可点击性、接口返回码是否一致。
- 重点核对“操作员不可删改重要数据”：系统配置修改、用户治理、审计导出、单据删除必须返回权限不足。
- 高风险动作必须有审计留痕：配置修改、迁移执行、用户重置密码、越权访问尝试。

## 6. 导航可见性原则（权限说明口径）

为避免“看得到但进不去”或“看不到但可直接访问”的体验偏差，权限说明统一采用以下导航可见性原则：

1. **同源原则**：菜单可见性与路由权限源自同一套权限点，不允许页面私自维护一套独立白名单。
2. **最小暴露原则**：仅展示当前角色可执行核心动作的入口，不展示高风险治理能力入口（如系统配置修改、用户治理、审计导出）。
3. **父子联动原则**：父级菜单是否显示由“至少一个可访问子路由”决定；若子路由全部不可访问，则父级自动隐藏。
4. **隐藏可达声明原则**：`menu:false` 路由可保留业务流程可达能力，但必须在文档中标注“不可见入口 + 可访问条件”。
5. **三层一致原则**：菜单可见、按钮可用、接口鉴权三层必须一致，不允许长期存在“前端放行后端 403”或“前端拦截后端放行”。
6. **回退可恢复原则**：未授权访问必须回退到该角色首个可访问页，避免空白页、死循环或错误落到高风险页面。
