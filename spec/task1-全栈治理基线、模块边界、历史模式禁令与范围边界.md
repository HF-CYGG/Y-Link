# Task1：全栈治理基线、模块边界、历史模式禁令与范围边界

> 文档说明：本文档用于沉淀本轮企业级可维护性治理的统一基线，明确前后端模块职责、允许协作方式、历史模式禁令与整改范围边界。
> 适用范围：前端开发、后端开发、联调测试、代码评审、任务拆解与后续治理验收。
> 维护口径：本文档描述的是“本轮整改必须遵守的治理口径”，不是未来所有阶段的完整蓝图；若后续范围升级，必须先更新本文档再扩展实现。

## 1. Task1 目标与交付结论

### 1.1 本文档要解决什么问题
- 当前仓库已经具备前端、后端、权限、审计、数据治理与部署等多类能力，但规范主要分散在页面实现、服务实现、白皮书说明和历史交付物中，缺少一份可直接用于评审与执行的统一边界清单。
- Task1 的目标不是立刻完成所有治理开发，而是先把“本轮到底要治理什么、谁负责什么、哪些写法从现在开始禁止新增、哪些范围明确不做”说清楚。
- 因此，Task1 的核心交付物是统一口径，而不是部署链路改造或大规模代码替换。

### 1.2 Task1 已落地的统一结论
- 已建立覆盖前端共享层、后端服务层、权限点、审计链路与数据治理的统一治理基线。
- 已明确各层模块边界，要求共享层、页面层、路由层、服务层、实体层、脚本层各司其职，禁止继续交叉堆叠。
- 已沉淀历史模式禁令，后续新增需求与修复任务不得继续沿用旧的散点实现方式。
- 已明确本轮范围只覆盖“业务治理与可维护性治理”，不包含部署链路、容器编排和 `1Panel` 一键化改造。

## 2. 全栈治理基线

### 2.1 前端治理基线
- 路由层职责限定在入口编排、懒加载、权限元信息和页面预热，不承担业务计算、接口拼装与状态纠偏。
- 页面层职责限定在视图装配、交互编排与状态展示，复杂表单联动、请求稳态控制、设备适配策略必须优先沉淀到组合式能力或共享组件中。
- 共享层必须集中在 `src/components/common`、`src/composables`、`src/utils`、`src/api` 等公共目录，统一承接弹层壳、请求治理、错误归一化、权限判断和通用交互。
- 高频列表、筛选和详情刷新场景必须接入 `useStableRequest` 或等价的稳定请求治理能力，避免旧请求覆盖新结果、避免页面闪烁。
- 请求错误必须走 `src/utils/error.ts` 统一归一化，不允许页面随意拼接不一致的提示口径。

### 2.2 后端治理基线
- 路由层职责限定在参数接收、权限前置校验、调用服务和返回响应，不直接编排业务规则，不在路由中书写跨表核心逻辑。
- 服务层是核心业务规则唯一收口层，负责事务、状态机、数据校验、审计触发与领域规则组合。
- 涉及库存、订单、核销、系统配置、用户治理、数据维护等关键写操作时，必须优先使用事务保护，避免出现部分成功、部分失败的脏状态。
- 实体层与数据库约束层负责结构真实性，字段语义、空值策略、枚举边界与索引约束必须与服务规则一致，不能只靠前端兜底。
- 脚本层职责限定在验证、回归、数据修复与演练，不得替代正式业务接口成为长期入口。

### 2.3 权限与审计治理基线
- 页面是否展示入口、按钮是否可见、接口是否可执行必须保持同一套权限口径，禁止“前端可见但后端拒绝”或“前端隐藏但接口仍可直接调用”的长期分裂状态。
- 高危写操作至少需要记录操作者身份、请求上下文、目标对象、动作类型、结果摘要与必要的变更说明。
- 供货方、管理端管理员、客户端用户必须保持角色边界清晰，不能因为页面复用而放宽接口权限。
- 权限矩阵与审计口径属于治理基线的一部分，后续新增高危动作时必须先补权限点与审计要求，再交付业务功能。

### 2.4 数据治理基线
- 导入导出、系统配置、运行时配置、关键状态字段与跨表关系字段必须具备结构校验、默认值兜底和异常路径说明。
- 任何影响库存、订单状态、配置启停、用户身份或迁移状态的字段，都不能仅依赖前端格式校验，后端与数据库必须至少有一层强约束。
- 数据修复脚本只能用于受控演练、专项修复和验收辅助，不能替代常规治理流程。

## 3. 模块边界清单

### 3.1 前端模块边界
| 模块 | 目录范围 | 允许职责 | 禁止事项 |
| --- | --- | --- | --- |
| 路由入口层 | `src/router/**` | 懒加载、入口分组、角色元信息、预热配置 | 直接写页面业务规则、直接发业务请求、直接维护表单状态 |
| 共享组件层 | `src/components/common/**` | 通用壳、通用展示、通用工作台布局、通用弹层骨架 | 直接耦合订单/用户/配置等具体业务接口 |
| 组合式能力层 | `src/composables/**` | 请求治理、设备适配、页面复用逻辑、操作防抖与幂等 | 在多个页面重复复制逻辑、不经抽象直接操作特定页面 DOM |
| API 接入层 | `src/api/**` | 接口封装、类型对齐、统一调用出口 | 在页面内重复拼 URL、散点处理接口异常 |
| 页面实现层 | `src/views/**` | 页面布局、业务交互编排、组件装配 | 私自复制共享壳能力、绕开统一权限和错误治理 |
| 状态存储层 | `src/store/**` | 会话态、跨页面共享态、持久化缓存 | 塞入临时视图状态、直接承载复杂页面流程编排 |

### 3.2 后端模块边界
| 模块 | 目录范围 | 允许职责 | 禁止事项 |
| --- | --- | --- | --- |
| 路由层 | `backend/src/routes/**` | 参数整理、调用中间件、调用服务、统一返回 | 直接操作数据库、直接拼接复杂业务规则 |
| 服务层 | `backend/src/services/**` | 核心规则、事务处理、状态流转、审计写入 | 依赖 `req/res`、把规则散落回路由层 |
| 实体层 | `backend/src/entities/**` | 表结构定义、字段约束、关系声明 | 承担业务流程编排、放置临时脚本逻辑 |
| 中间件层 | `backend/src/middleware/**` | 认证、鉴权、错误兜底、请求上下文补充 | 承担具体业务规则判断 |
| 工具与常量层 | `backend/src/utils/**`、`backend/src/constants/**` | 通用工具、错误映射、请求上下文、权限常量 | 混入具体业务流程状态机 |
| 验证与修复脚本层 | `backend/scripts/**` | 专项验证、演练、数据修复、回归校验 | 替代正式接口、绕过正常治理流程长期运行 |

### 3.3 跨端协作边界
- 前端通过 `src/api/modules/**` 与后端接口交互，不直接约定隐藏字段或临时响应结构。
- 后端新增字段、状态或权限点时，必须同步更新前端类型、显示口径和必要文档，不能把变更留在口头约定。
- 审计、权限、配置等治理能力属于全栈共识能力，任何单端调整都必须同步检查另一端是否需要配套修正。

## 4. 历史模式禁令

### 4.1 前端禁止继续新增的历史模式
- 禁止新增裸 `el-dialog`、裸 `el-drawer` 直接堆业务内容，而不经过共享壳能力收口。
- 禁止在页面中散点编写重复请求加载、重复错误提示、重复权限判断与重复空状态逻辑。
- 禁止高频列表直接 `await` 普通请求后覆盖状态，而不做旧请求拦截与竞态治理。
- 禁止在页面样式中继续堆叠魔法数字修补高度、滚动与布局问题，必须优先回到共享壳或布局基座治理。
- 禁止把系统指令、调试标签、内部治理术语直接暴露到最终用户界面。

### 4.2 后端禁止继续新增的历史模式
- 禁止在路由文件中直接拼装核心事务逻辑、跨表更新逻辑与状态机判断。
- 禁止关键写接口只做业务成功返回，不补操作者、上下文与结果留痕。
- 禁止把权限点做成过粗粒度的“超级权限”后在多个高危动作中复用，导致授权失真。
- 禁止把数据修复脚本当成常规业务入口长期依赖，绕开正式接口与审计链路。
- 禁止只靠前端约束敏感字段格式，后端与数据库层没有任何兜底校验。

### 4.3 文档与协作禁令
- 禁止新增功能只改代码不改文档，导致模块边界再次依赖口头传递。
- 禁止把“未来计划”和“当前已实现事实”混写在同一段描述中而不标注状态。
- 禁止在未更新治理口径前，先扩展与既有边界冲突的新模式。

## 5. 范围边界

### 5.1 本轮明确纳入的范围
- 前端共享模式治理，包括弹层壳、列表请求、错误提示、按钮权限与页面装配规范。
- 后端服务治理，包括路由与服务职责收口、关键写操作审计、事务边界与权限细化。
- 数据治理，包括系统配置、运行时配置、导入导出与关键字段完整性约束。
- 文档治理，包括模块职责、权限矩阵、回归清单与历史模式禁令沉淀。

### 5.2 本轮明确不纳入的范围
- 不包含 `Docker Compose`、`Dockerfile`、`Nginx`、`onebox`、`1Panel` 一键化和发布流水线改造。
- 不包含服务器部署拓扑、反向代理策略、镜像发布策略与运维面板自动化能力升级。
- 不包含与本轮治理目标无关的全量 UI 重设计或业务流程重构。

### 5.3 为什么要先切清范围
- 当前项目已经同时覆盖管理端、客户端、后端、数据库与部署能力，如果本轮把业务治理和部署治理绑在一起，任务会过大且验收失焦。
- 先完成业务治理基线，可以为后续部署治理提供更稳定的接口、配置口径和文档基础。
- 因此，本轮范围边界不是“永久不做”，而是“本阶段明确拆项，避免互相拖累”。
