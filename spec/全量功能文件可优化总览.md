# 全量功能文件可优化总览

## 1. 文档目的
- 本文档用于沉淀对 `f:\Y-Link` 功能文件的新一轮逐文件巡检结果。
- 本轮目标不是直接改代码，而是确保所有纳入范围的功能文件都被逐个查看，并将可优化点整理成后续可执行主清单。
- 本文档同时区分“历史已知项”“本轮新发现项”“低风险快速优化”“中长期结构优化”，避免后续排期时混淆优先级。

## 2. 巡检范围与方法
### 2.1 巡检原则
- 逐个文件阅读，不使用抽样式结论替代文件级结论。
- 以前端功能页面和后端功能链路为核心，必要时追到 API 模块和路由装配层确认前后端映射。
- 每条优化结论都要求能够落到具体文件，而不是只给泛化建议。

### 2.2 本轮纳入范围
- `src/views/**`：53 个
- `backend/src/routes/**`：14 个
- `backend/src/services/**`：19 个
- `src/api/modules/**`：15 个
- `src/router/**`：3 个
- 合计：104 个功能相关文件

### 2.3 本轮关注重点
- 页面文件是否过重、是否存在职责混叠、是否存在交互状态与提交状态不一致。
- 前后端接口契约是否漂移、是否存在字段口径不一致、前端封装缺失或权限口径冲突。
- 服务层是否存在权限绕过、性能退化、单机内存态风险、缺少服务层兜底校验等问题。
- 路由与预热策略是否仍有冗余带宽消耗或角色不匹配问题。

## 3. 整体结论
- 项目当前并不存在“大面积失控”问题，主链路整体可用，最近几轮治理已经显著改善了性能、共享层和权限体系。
- 现阶段最值得继续优化的方向，不是继续铺大范围重构，而是围绕 4 类问题持续收口：
  - 高风险功能断层与越权风险
  - 前后端契约漂移与漏封装
  - 单文件职责过重导致的维护成本
  - 低频重能力和预热策略仍可继续压缩
- 本轮巡检识别出一批“改动不大但收益直接”的低风险项，也识别出若干需要专项推进的结构性问题。

## 4. 高优先级问题
### 4.1 客户端结算页采集了提货人，但未真正提交
- 文件：`src/views/client/ClientCheckoutView.vue`
- 现象：页面维护了 `pickupContact`，并参与展示与本地记忆，但提交预订单 payload 时未带出该字段。`[已解决：前端提交链路已携带 pickupContact，并补齐后端存储与详情回传字段 pickup_contact]`
- 根因：前端需求已经扩展到“提货人可编辑”，但接口入参与后端消费逻辑未同步完成。
- 风险：用户可见输入与实际落库信息不一致，属于功能闭环缺口。
- 建议：先统一前后端字段定义，再把该字段纳入结算提交、订单详情展示和后端存储。

### 4.2 O2O 核销页页面权限与实际提交权限口径冲突
- 文件：`src/router/routes.ts`、`backend/src/routes/o2o.routes.ts`
- 现象：前端允许 `orders:view` 进入核销页，但后端核销动作依赖 `orders:create`。`[已解决：核销动作统一收敛到 orders:create，退货拒绝/现场改单统一收敛到 orders:update]`
- 根因：页面访问权限与关键动作权限分离后，没有继续统一成同一业务语义权限。
- 风险：用户可以进入页面但关键动作失败，形成“可见不可用”的功能断层。
- 建议：统一核销台所需权限口径，优先收敛成业务语义更明确的独立权限点。

### 4.3 入库核销服务缺少服务层门禁
- 文件：`backend/src/services/inbound.service.ts`
- 现象：`verifyInbound` 负责改库存与改单据状态，但服务层自身未校验操作者角色。`[已解决：服务层已补 admin/operator 角色兜底校验]`
- 根因：当前主要依赖路由层门禁，服务层缺少二次兜底。
- 风险：若后续有新路由或脚本复用该方法，存在权限绕过风险。
- 建议：在服务层引入 actor 级角色/权限校验，并让路由层与服务层共同守住边界。

### 4.4 标签服务层校验过薄
- 文件：`backend/src/services/tag.service.ts`
- 现象：创建/更新/删除标签时，服务层对空值、唯一冲突、引用保护的兜底不够完整。`[已解决：已补名称/编码唯一校验、关联引用阻断与业务化错误提示]`
- 根因：标签服务目前更像数据库薄封装，规则校验主要依赖前端或底层异常。
- 风险：容易形成重复标签、空标签、删除被引用标签时报数据库异常。
- 建议：把业务规则上提到服务层，形成统一校验与业务化错误文案。

### 4.5 客户端认证与找回密码链路风控接入不完整
- 文件：`backend/src/services/client-auth.service.ts`、`backend/src/services/auth-security.service.ts`
- 现象：注册、找回密码等入口未完整接入现有风控守卫。`[已解决：已接入认证链路频控守卫，并统一票据内存存储清理策略]`
- 根因：风控能力已在共享安全服务中存在，但业务入口接入不彻底。
- 风险：注册、找回密码链路更容易被高频尝试、枚举或刷接口。
- 建议：把客户端认证全链路统一接入风控与限流，并补票据过期清理。

## 5. 中优先级问题
### 5.1 前端存在若干契约漂移和漏封装
- 文件：`src/api/modules/inbound.ts`、`backend/src/routes/inbound.routes.ts`
- 问题：后端有 `/inbound/admin/list`，前端 API 模块没有对应封装。`[已解决：已补齐管理端列表封装并对齐查询参数类型]`
- 文件：`src/api/modules/o2o.ts`、`backend/src/routes/o2o.routes.ts`
- 问题：后端支持 `startTime/endTime` 查询，前端类型仅保留 `status/keyword/limit`。`[已解决：已补齐 startTime/endTime 与正式查询参数类型]`
- 文件：`src/api/modules/client-auth.ts`、`src/api/modules/client-user-manage.ts`、`backend/src/routes/client-auth.routes.ts`、`backend/src/routes/client-user-manage.routes.ts`
- 问题：`username` 与 `realName` 命名口径混用。`[已解决：前后端已统一优先使用 username，并保留 realName 兼容别名]`
- 建议：用接口契约对齐专项统一梳理前后端字段、筛选项和缺失封装。

### 5.2 若干核心页面仍然单文件过重
- 文件：`src/views/client/ClientOrderDetailView.vue`
- 问题：单文件同时承载详情展示、改单、退货、二维码、打印、PDF 导出、列表缓存同步。`[已解决：已完成第一轮拆分，改单/退货/出库单弹层下沉为独立组件]`
- 文件：`src/views/system/SystemConfigView.vue`
- 问题：同页叠加订单流水号、O2O 规则、验证码通道、客户端部门配置等多个配置域。`[已解决：已完成第一轮拆分，按配置域下沉为分区组件]`
- 文件：`src/views/system/DatabaseMigrationView.vue`
- 问题：数据库迁移编排、步骤状态、执行结果和页面交互耦合在单页。`[已解决：已完成第一轮拆分，总览/步骤/预检/执行结果面板下沉]`
- 文件：`src/views/o2o/O2oVerifyConsoleView.vue`
- 问题：预订单核销、退货拒绝、现场改单、合规状态等分支全部堆叠在一个页面。`[已解决：规则逻辑已下沉 helper，并用稳定请求收敛并发覆盖]`
- 建议：按功能域拆子组件和 composable，先拆最重、最常改的部分。

### 5.3 数据加载策略仍有可继续优化空间
- 文件：`src/views/inbound/SupplierHistoryView.vue`
- 问题：先拉全量再做前端筛选，数据量增长后会放大首屏与内存压力。`[已解决：已迁移为服务端筛选+分页]`
- 文件：`backend/src/services/dashboard.service.ts`
- 问题：首页聚合查询串行执行，未来数据增长时会拉高工作台首屏耗时。`[已解决：关键统计与排行查询已并行化]`
- 文件：`backend/src/services/order.service.ts`
- 问题：多字段 `%LIKE%` 搜索在订单量增大后难以走索引。`[已解决：已增加 showNo 精确匹配分支，优先命中更窄查询]`
- 文件：`backend/src/services/product.service.ts`
- 问题：产品列表仍走全量返回，未分页。`[已解决：已新增分页查询能力 /products/paged 并支持 page/pageSize]`
- 建议：把“前端轻筛选”页面逐步迁移到服务端筛选/分页，并补慢查询基线。

### 5.4 单机内存态票据与验证码存储需要继续治理
- 文件：`backend/src/services/captcha.service.ts`
- 文件：`backend/src/services/client-auth.service.ts`
- 文件：`backend/src/services/verification-code.service.ts`
- 文件：`backend/src/services/auth-security.service.ts`
- 问题：多类验证码/风控/重置票据仍主要依赖进程内 `Map` 或内存状态，缺少统一 TTL 清理与多实例方案。`[已解决：已引入统一 EphemeralTicketStore，补齐 TTL 清理和容量控制]`
- 风险：多实例不一致、进程重启丢失、长运行内存增长。
- 建议：至少先补统一过期回收；若后续进入多实例部署，再迁到共享存储。
