# Y-Link 验证脚本与高风险操作

## 适用范围

- 适用于选择验证脚本、判断文档/配置/代码改动最少要跑哪些验证，以及识别高风险操作边界。
- 适用于提交前和交付前的验证决策。

## 最后核对来源

- `package.json`
- `backend/package.json`
- `AGENTS.md`
- `backend/src/services/database-migration.service.ts`
- `backend/src/services/data-maintenance.service.ts`
- `backend/src/routes/o2o.routes.ts`
- `backend/src/routes/inbound.routes.ts`

## 真实入口

- 前端/根验证脚本：`package.json` 的 `scripts`
- 后端验证脚本：`backend/package.json` 的 `scripts`
- 高风险动作门禁：永久删除密码、管理员角色、迁移/导入导出接口权限

## 前端链路

- 文档、中文文本或规则文件改动后，最低应跑 `npm run verify:text-encoding`。
- 前端核心改动后，最低应跑 `npm run build`。
- 大版本、路由重构、核心链路重构后，应跑 `npm run verify:performance`。

## 后端链路

- 后端 TypeScript 改动后，应跑 `npm --prefix backend run build`。
- 权限、路由、鉴权、安全边界改动后，应补充：
  - `npm --prefix backend run task2:route-contract:verify`
  - `npm --prefix backend run security:hardening:verify`
- O2O、订单、入库、通知等已有专项验证脚本时，应优先使用对应脚本而不是只跑构建。

## 关键状态/字段/快照

- 高风险操作包括但不限于：
  - 永久删除订单或送货单
  - 数据库迁移与运行时切库
  - JSON 导入导出
  - 通知外发与验证码通道修改
  - 影响库存、订单状态、客服消息流转的服务层改动
- 这些场景除了通过权限，还应在说明中明确写出风险面和验证结果。

## 权限与安全边界

- 永久删除类操作不仅要有权限，通常还要求永久删除密码。
- 数据库迁移、运行时覆盖切换、系统级配置变更通常要求管理员角色。
- 通知与验证码通道配置属于外部联动安全边界，不应视为普通表单改动。

## 常见异常与排查顺序

1. 文档改了但字符乱码：先跑 `npm run verify:text-encoding`。
2. 前端看似没问题但构建挂了：跑 `npm run build`，重点看路由和类型引用。
3. 后端接口能跑但权限边界错了：跑后端路由契约和安全验证。
4. 性能回归没被发现：检查是否遗漏 `npm run verify:performance`。
5. 高风险操作说明不完整：核对是否单独说明了数据库、通知、订单、库存或权限风险。

## 验证与回归关注点

- 文档/规则改动：`npm run verify:text-encoding`
- 前端结构改动：`npm run build`
- 后端结构改动：`npm --prefix backend run build`
- 权限/路由/安全改动：后端契约与安全验证脚本
- 大型重构：性能验证
