# Y-Link 客户端鉴权、商城、购物车与结算

## 适用范围

- 适用于客户端登录、注册、找回密码、商城浏览、购物车、结算和个人中心场景。
- 适用于定位“刷新后为什么还能登录、购物车为什么会保留、换号为什么要清缓存”的问题。

## 真实入口

- 页面入口：`/client/login`、`/client/mall`、`/client/cart`、`/client/checkout`、`/client/profile`
- 前端状态真源：`client-auth.ts`、`client-cart.ts`
- 后端鉴权真源：`client-auth.middleware.ts`、`client-auth.service.ts`

## 前端链路

- `client-auth.ts` 启动时先读本地快照，再调用 `/client-auth/me` 校验服务端 Cookie 会话是否仍有效。
- 登录、注册、找回密码、资料更新统一通过客户端鉴权 Store 收口。
- 购物车 Store 在本地持久化的不只是商品和数量，还包括价格、库存、限购和选中态。
- 商城和购物车展示共用 O2O 价格口径工具，避免页面各自计算折扣价。
- 结算页提交成功后通常跳订单详情，因此客户端路由预热把订单详情放在高优先级。

## 后端链路

- `client-auth.middleware.ts` 优先从客户端安全 Cookie 读取会话，再兼容 Bearer。
- 客户端会话失效时，服务端直接返回未登录，由前端 Store 清理本地快照。
- 商城商品来自 O2O 商品接口，而不是单独的客户端商品表。

## 关键状态/字段/快照

- 客户端鉴权快照核心字段：`currentUser`、`expiresAt`、`initialized`。
- 购物车约束统一用 `toMaxQty()`，即 `availableStock` 与 `limitPerUser` 的合并口径。
- 购物车在目录刷新后会执行 `syncWithCatalog()`，尽量保留用户选择但刷新库存和价格。

## 权限与安全边界

- 客户端登录态不应由本地缓存单独判定，必须以后端 `/client-auth/me` 校验结果为准。
- 退出登录和账号切换时必须清理购物车、目录、订单等用户绑定缓存，防止串号。
- 客户端页面不应绕过 Store 直接判断 Cookie 是否存在。

## 常见异常与排查顺序

1. 刷新后自动掉线：查 `/client-auth/me` 是否失败，再查本地快照是否被清空。
2. 换账号后购物车串号：查 `clearClientScopedStores()` 是否执行。
3. 购物车数量异常：查 `toMaxQty()` 和商品目录最新库存。
4. 商城价格和购物车价格不一致：查是否统一使用 O2O 价格工具。
5. 结算后没跳正确页面：查路由预热和结算成功后的跳转逻辑。

## 验证与回归关注点

- 客户端鉴权回归：登录、刷新恢复、退出登录、找回密码。
- 商城链路回归：加购、改数量、清空、切换账号、刷新后恢复。
- 结算链路回归：下单前校验、提交成功跳转、库存不足提示。
