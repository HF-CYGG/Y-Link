# Y-Link 本地联调与部署模式

## 适用范围

- 适用于本地联调、onebox、双容器云端部署和外置 MySQL 部署模式的区分。
- 适用于定位“当前机器为什么连到这个数据库、哪个容器负责前端/后端/上传目录”的问题。

## 最后核对来源

- `start-local-dev.ps1`
- `compose.cloud.yml`
- `compose.mysql.yml`
- `Dockerfile.onebox`
- `docker/nginx/onebox.conf`

## 真实入口

- 本地联调脚本：`start-local-dev.ps1`
- SQLite/双容器云端部署：`compose.cloud.yml`
- 外置 MySQL 双容器部署：`compose.mysql.yml`
- 单容器 onebox：`Dockerfile.onebox`

## 前端链路

- 本地联调只启动一个 Vite 开发服务器，同时承载管理端和客户端页面。
- onebox 模式下前端静态资源由 Nginx 提供，并代理后端接口与健康检查。
- 双容器模式下前端容器的健康检查同时覆盖静态页和 `/health` 代理链路。

## 后端链路

- 本地联调脚本会：
  - 清理旧 PID 和端口占用
  - 生成本次后端临时 env
  - 拉起前后端
  - 等待 `/health` 和前端端口就绪
  - 输出当前实际数据库摘要与来源
- `compose.cloud.yml` 默认仍可使用 SQLite，并要求挂载 `/app/data` 与 `/app/uploads`。
- `compose.mysql.yml` 只起应用容器，数据库由外置 MySQL 提供。
- `Dockerfile.onebox` 在同一容器内打包前端、后端与 Nginx。

## 关键状态/字段/快照

- 本地联调的数据库摘要以 `/health` 返回为准，不以 env 猜测为准。
- onebox 和 cloud compose 都需要考虑上传目录持久化，否则图片会随容器丢失。
- 运行时数据库覆盖存在时，部署模式和 `.env` 看到的值不一定等于真实生效数据库。

## 权限与安全边界

- 部署文档中的环境变量不应写入真实密码。
- onebox 和 compose 都依赖后端对敏感配置的脱敏日志策略。
- 外部服务和 Webhook 仍由后端安全校验，而不是部署层兜底。

## 常见异常与排查顺序

1. 本地启动成功但页面数据不对：先看联调脚本输出的数据库摘要和 `/health`。
2. 线上图片 404：查 `/app/uploads` 是否持久化挂载。
3. 双容器前端健康检查失败：查前端是否既能访问根路径又能代理 `/health`。
4. onebox 能打开首页但接口异常：查 Nginx 代理和后端启动是否都成功。
5. 切到 MySQL 后行为还像 SQLite：查运行时覆盖和实际 `/health` 摘要。

## 验证与回归关注点

- 改本地联调脚本后回归：启动、停止、日志输出、数据库摘要、前端地址。
- 改 compose/onebox 后回归：容器健康检查、图片访问、管理端登录、客户端商城。
