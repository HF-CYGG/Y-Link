# Y-Link 后端启动、路由装配与健康检查

## 适用范围

- 适用于后端启动失败、健康检查异常、接口挂载顺序疑问和数据库模式判断。
- 适用于回答“后端启动时到底先做了什么、`/health` 为什么是判断真相的第一入口”。

## 最后核对来源

- `backend/src/index.ts`
- `backend/src/app.ts`
- `backend/src/config/data-source.ts`
- `backend/src/config/database-bootstrap.ts`
- `backend/src/config/database-runtime-override.ts`

## 真实入口

- 启动入口：`backend/src/index.ts`
- 应用装配入口：`backend/src/app.ts`
- 数据源入口：`backend/src/config/data-source.ts`

## 前端链路

- 管理端和本地联调脚本都依赖 `/health` 判断后端是否真正起来。
- 前端不应通过猜测环境变量判断数据库模式，而是应信任 `/health` 返回的 effective database 摘要。

## 后端链路

- 启动顺序大致为：
  1. 准备运行时数据库上下文
  2. 初始化数据源
  3. 安装 SQLite 事务队列
  4. 初始化数据库结构
  5. 迁移旧上传引用
  6. 确保默认管理员
  7. 确保默认系统配置
  8. 确保默认通知规则
  9. 启动 HTTP 服务并执行启动自检
- `app.ts` 先挂公共安全头，再挂上传静态资源，再挂公开路由，最后统一给受保护 `/api` 接口加管理端鉴权链路。

## 关键状态/字段/快照

- `/health` 返回：`status`、effective database 摘要、运行时覆盖状态。
- 真正生效的数据库模式可能来自环境变量，也可能来自运行时覆盖配置。
- SQLite 模式下会安装事务队列以降低锁冲突。

## 权限与安全边界

- 公开路由只包括少数认证/O2O/客户端反馈入口。
- 其余管理端 `/api` 请求统一先过 `requireAuth` 与 `requireAdminCsrf`。
- 启动日志不会输出敏感明文密码，只给脱敏摘要。

## 常见异常与排查顺序

1. 服务起不来：先看 `backend/src/index.ts` 的启动阶段日志卡在哪一环。
2. `/health` 可访问但业务不对：看 effective database 摘要是否连到了错误数据库。
3. 本地和线上表现不一致：对比 `app.ts` 的路由挂载顺序与部署方式。
4. SQLite 模式偶发锁问题：先确认事务队列是否安装，再看业务事务是否过大。

## 验证与回归关注点

- 改启动或装配逻辑后至少回归：后端可启动、`/health` 返回正常、管理端登录和客户端匿名商品接口正常。
- 改数据库模式相关逻辑后回归：`/health` 摘要、运行时覆盖状态、对应验证脚本。
