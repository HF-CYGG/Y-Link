# Y-Link 入库、供应方与库存核验

## 适用范围

- 适用于供应方送货单、改单、撤销、删除、恢复、永久删除、扫码入库和管理端入库核验场景。
- 适用于查“供应方为什么能看这个单、为什么这个送货单不能改、为什么扫码后库存变了”的问题。

## 最后核对来源

- `src/views/inbound/*`
- `src/api/modules/inbound.ts`
- `backend/src/routes/inbound.routes.ts`
- `backend/src/services/inbound.service.ts`
- `backend/src/entities/biz-inbound-order.entity.ts`
- `backend/src/entities/biz-inbound-order-item.entity.ts`

## 真实入口

- 供应方入口：`/supplier-delivery`、`/supplier-history`
- 管理端入口：`/inbound-scan`、`/inbound-manage`
- 后端服务真源：`inbound.service.ts`

## 前端链路

- 供应方通过共享工作台壳层在“录入”和“历史”之间切换。
- 管理端通过扫码页和入库管理页执行核验与现场改单。
- 前端会根据 showNo、verifyCode、状态和删除标记组合请求详情。

## 后端链路

- `inbound.routes.ts` 明确拆成两套入口：
  - 供应方接口：提交、列表、改单、撤销、软删、恢复、永久删、按 showNo / verifyCode 查详情
  - 库管接口：核验入库、现场改单、查看全部送货单
- `inbound.service.ts` 负责：
  - 供应方送货单提交流程
  - 仅允许供应方修改自己的可变订单
  - 状态流转：待入库、已核验、已撤销、已删除
  - 管理端核验入库与库存回写
  - 供应方和管理端的可见性边界

## 关键状态/字段/快照

- 关键字段：`showNo`、`verifyCode`、`status`、`supplierId`、`isDeleted`、`items`。
- 供应方改单/撤销只能操作未核验且归属自己的单据。
- 永久删除是对已软删单据的第二阶段治理，不等于普通删除。
- 核验入库会实际回写库存，不只是把送货单状态改成“已入库”。

## 权限与安全边界

- 供应方侧依赖 `inbound:create`、`inbound:view`。
- 管理端核验依赖 `inbound:verify`，查看全量列表通常要求 `inbound:view` 且角色为 `admin/operator`。
- 供应方即便有 `inbound:view`，服务层仍会按 supplier 身份限制可见范围，防止参数探测越权。
- 永久删除还要求永久删除密码。

## 常见异常与排查顺序

1. 供应方看到了别人的单：先查服务层是否按 `supplierId` 收口，而不是只看路由权限。
2. 单据不能改/不能撤销：查当前状态是否已核验、已撤销或已删除。
3. 扫码后库存未变：查核验逻辑是否真正落库，而不是只更新状态。
4. showNo/verifyCode 查询不到：查删除标记和角色可见范围。
5. 管理端能看列表但点详情失败：查路由权限、详情入口参数和服务层二次校验。

## 验证与回归关注点

- 供应方链路至少回归：新建、列表、改单、撤销、软删、恢复、永久删。
- 管理端链路至少回归：扫码核验、现场改单、全量列表、详情查看。
- 涉及删除治理时回归：showNo 二次确认、永久删除密码、状态变化与库存一致性。
