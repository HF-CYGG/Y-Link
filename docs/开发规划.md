# Y-Link 线上预订与线下核销系统开发规划

> 文档说明：本文用于沉淀 O2O 线上预订与线下核销能力的阶段性开发规划、模型设计与实施路线。
> 适用范围：产品规划评审、技术方案拆解、阶段排期沟通与里程碑验收。
> 维护口径：本文属于规划文档，部分章节描述的是目标态而非当前已上线事实；执行开发时必须同步标注已完成项与未完成项，避免规划与现网状态混淆。

## 1. 架构演进与技术栈选择

为了满足面向普通用户的线上预订需求，同时兼顾管理端的出入库操作，系统将从现有的单端（PC 管理端）演进为双端架构。

*   **客户端 H5 (面向各部门订购人)**：
    *   **技术栈**：Vue 3 (Composition API) + TypeScript + Vite + **Vant 4** (或 NutUI，专为移动端优化的 Vue 组件库) + Tailwind CSS。
    *   **设计语言**：延续全局规范的“具象化极简”与 Bauhaus 风格。采用大圆角、大留白、卡片化布局，消除传统后台的边框感，提供类似现代新零售电商（如喜茶、瑞幸）的流畅体验。
*   **管理端 PC (面向工作人员)**：
    *   **技术栈**：维持现有 Vue 3 + Element Plus 架构。
*   **后端与数据库 (平滑迁移准备)**：
    *   **ORM 层**：继续深化 TypeORM 的使用。TypeORM 原生支持 SQLite 与 MySQL 的无缝切换。
    *   **MySQL 预留**：通过环境变量（如 `DB_TYPE=sqlite` 或 `mysql`）控制数据库连接驱动。所有实体类（Entity）避免使用特定数据库的专有字段类型，统一使用通用类型（如 `varchar`, `int`, `boolean`）。
    *   **并发控制**：在 SQLite 阶段，通过应用层重试机制和数据库乐观锁处理并发写入；切换至 MySQL 后，利用 InnoDB 的行级锁天然解决高并发库存扣减问题。

## 2. 数据库模型扩展设计

为了实现“实时库存扣减”与“预订展示”，需要对核心库存逻辑进行重构。

### 2.1 库存双字段模型 (重点)
在 `Product` 表中，拆分库存概念：
*   **`current_stock` (实际物理库存)**：仓库里真实存在的商品数量。**仅由管理端入库/出库操作修改**。
*   **`pre_ordered_stock` (已预订数量)**：已被客户线上预订，但尚未线下取货的商品数量。**由客户端下单操作增加，管理端核销操作扣减**。
*   **客户端展示逻辑**：
    *   `剩余可用数量` = `current_stock` - `pre_ordered_stock`。
    *   `已被预订数量` = `pre_ordered_stock`。
    *   当 `剩余可用数量 <= 0` 时，客户端显示售罄，不可继续预订。

### 2.2 新增与扩展核心实体
*   **Product (商品表扩展)**：新增 `status` (上架/下架状态，默认下架)、`thumbnail` (预览图 URL)、`detail_content` (富文本详情)、`limit_per_user` (单次/单人限购数量，防恶意囤积)。
*   **ClientUser (客户端用户)**：字段包含 `手机号` (唯一索引)、`密码/哈希`、`所属部门`、`真实姓名`、`注册时间` 等。
*   **PreOrder (预订单)**：字段包含 `订单号` (连续流水或雪花算法)、`客户端用户ID`、`核销凭证码` (UUID)、`订单状态` (待取货、已核销、已取消)、`总计件数`。
*   **PreOrderItem (预订单明细)**：关联预订单与商品，记录下单时的预定数量。
*   **InventoryLog (出入库流水)**：记录所有的物理库存变动，区分操作类型（如：采购入库、预订出库、散客出库）。

## 3. 核心业务流程与页面布局

### 3.1 客户端 H5 页面规划 (移动端优先)

1.  **注册与登录页**：
    *   **UI 布局**：全屏纯色背景，居中展示品牌 Logo。提供“手机号 + 验证码”快捷登录/注册，并提供“忘记密码”入口。
    *   **验证码双重防刷机制**：封装 `SmsService` 预留阿里云/腾讯云 SMS 接口。**若系统暂未配置短信秘钥，自动降级使用图形验证码（基于 `svg-captcha` 库实现）**，结合后端 IP 频率限制，有效防止机器人批量注册。
2.  **密码找回页 (忘记密码)**：
    *   **流程设计**：包含“身份验证”与“重置密码”两个步骤。
    *   **验证机制**：用户输入已绑定的手机号并获取验证码（短信或降级图形验证码），校验通过后允许设置新密码，保障账号安全。
3.  **商品大厅 (首页)**：
    *   **UI 布局**：顶部为搜索栏；左侧或顶部为滑动分类导航；主体为商品信息流卡片。
    *   **卡片设计**：左侧商品主图，右侧包含标题。重点在右下角高亮显示 `当前剩余: X 件`，辅以浅灰色小字显示 `已被预订: Y 件`。
    *   **交互**：点击右侧 [+] 按钮直接加入底部悬浮的“待结算购物车”。
4.  **预订单提交页**：
    *   **UI 布局**：卡片式罗列所选商品及数量。底部表单自动带入用户的“姓名”、“手机号”、“部门”。
    *   **防超卖交互**：点击“提交订单”时，后端开启事务进行乐观锁校验。若失败，提示用户库存已被抢光。
5.  **订单详情与核销码页**：
    *   **UI 布局**：屏幕中央生成大尺寸、高对比度的 **二维码**（基于 `qrcode.vue` 前端生成，内容为订单的加密凭证串）。
    *   **辅助信息**：下方高亮提示“请在取货时向工作人员出示此二维码”，并列出预订明细与下单时间。

### 3.2 管理端 PC 页面规划

1.  **预订单核销台 (O2O 闭环核心)**：
    *   **隐式扫码交互**：页面无需专门的输入框，直接全局监听键盘的 `keyup` 事件。由于实体扫码枪本质上是模拟键盘连续输入并在末尾附加 `Enter` 键，系统监听到回车后，自动触发查询。
    *   **核销弹窗**：扫码成功后，右侧弹出 Drawer（抽屉），展示该预订单的详细信息及对应客户。
    *   **确认出库**：工作人员核对实物后，点击“确认出库”。此时后端执行：`current_stock` 和 `pre_ordered_stock` 同时减去订单数量，记录出库流水，订单状态变为“已核销”。
2.  **实物入库管理页**：
    *   **UI 布局**：新增专门的“入库管理”路由。
    *   **操作流程**：工作人员选择商品（支持扫商品本身的条码快速带出），输入入库数量，选择供应商。确认后，直接增加 `current_stock` 并生成入库流水。
3.  **商品大厅维护页 (全新功能)**：
    *   **上/下架管理**：商品列表页增加直观的“一键上/下架”状态开关，仅上架商品在客户端大厅可见。
    *   **预览图与详情编辑**：提供侧边抽屉或独立编辑页，支持上传图片作为“商品预览图 (thumbnail)”；集成富文本编辑器（如 WangEditor 或 Quill），供管理员录入图文并茂的商品详情内容，提升用户体验。
4.  **O2O 业务规则配置页 (扩展现有系统配置)**：
    *   **功能说明**：在现有系统配置页面中增加“线上预订设置”专属模块。
    *   **配置项**：允许管理员在此页面通过直观的开关和数值输入框灵活调整各项策略，如“订单超时自动取消”与“限购防囤积”的参数（详情见下文 3.3）。

### 3.3 高级优化特性 (参考成熟电商/O2O系统)

1.  **订单超时自动取消 (库存释放)**：
    *   预订单不能永久锁定库存。引入定时任务（Cron Job）或延时队列，若订单超过设定的保留期未被线下核销，系统自动将其状态置为“已取消”，并**释放已预订库存**（`pre_ordered_stock` 减去对应数量），避免恶意占库存。
    *   **管理端灵活配置**：该功能交由管理端进行配置。**系统默认开启该策略，且默认将超时取消时长自动设置为合理的 24 小时**。管理员可随时在后台关闭或修改时长。
2.  **限购防囤积策略**：
    *   针对紧俏商品，结合 `Product` 表扩展的 `limit_per_user` 字段。提交订单时校验该用户或该部门近期的购买流水，保障系统资源的公平分配。
    *   **管理端灵活配置**：同样支持由管理端随时开启或关闭。**系统默认开启全局限购，并自动设置全局默认合理值（例如：单人单次最多 5 件）**。若某件商品极度稀缺，管理员也可在商品编辑页为其设置更严格的独立限购数量以覆盖全局默认值。
3.  **大厅动态公告栏**：
    *   客户端首页增加跑马灯公告区域，便于管理员发布“缺货通知”、“到货预告”或“节假日领取说明”。

## 4. 数据备份与导入恢复机制

为保证数据安全以及未来向 MySQL 的平滑迁移，开发一套独立的数据维护模块：

1.  **全量物理备份 (针对 SQLite 阶段)**：
    *   提供系统接口，直接复制后端的 `.sqlite` 物理数据库文件，生成带时间戳的备份文件并提供下载。
2.  **跨数据库标准备份 (JSON 格式导出/导入)**：
    *   **导出**：读取各表的所有核心业务数据，组装成结构化的 JSON 文件下载。此格式与底层数据库类型无关。
    *   **导入恢复**：上传 JSON 文件，后端解析后，按依赖顺序（先用户/商品，后订单/流水）执行 `upsert`（存在则更新，不存在则插入）。
    *   **迁移策略**：在切换到 MySQL 时，只需先从 SQLite 导出 JSON，修改 `.env` 中的数据库配置重启服务，再将 JSON 导入空库即可完成数据无损迁移。

## 5. 阶段开发排期

*   **Phase 1：底层改造与 API 预置**
    *   修改实体模型（引入双库存字段），配置 TypeORM 以支持数据库灵活切换。
    *   开发数据备份与 JSON 导入导出通用接口。
    *   开发客户端用户认证、短信验证码模块 API。
*   **Phase 2：客户端 H5 开发**
    *   初始化移动端 Vue 3 项目结构。
    *   完成注册登录、商品大厅（实时库存计算与展示）、购物车与下单防超卖逻辑。
    *   生成订单详情及动态二维码页面。
*   **Phase 3：管理端闭环与出入库**
    *   开发“实物入库管理”模块。
    *   开发“核销工作台”，接入扫码枪全局事件监听，实现扫码扣库存的闭环。
*   **Phase 4：并发测试与验收**
    *   模拟多人同时下单，验证并发库存安全性。
    *   进行端到端的全流程演练。
