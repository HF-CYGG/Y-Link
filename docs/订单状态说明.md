# 订单状态说明（O2O 预订单与退货申请）

> 文档说明：本文用于统一客户端、管理端与核销台对 O2O 预订单、退货申请和门店核销动作的解释口径，避免同一状态被不同页面误读。
> 适用范围：订单列表、订单详情、退货申请、核销台、客服答疑、运营培训与接口联调。
> 维护口径：主状态、退货申请状态、状态报告字段与商家特殊状态必须与前后端枚举保持一致；新增状态时必须同步更新本文与代码常量。

## 1. 文档目的
- 统一客户端、管理端、核销台对“订单状态”的认知口径。
- 说明当前系统已实现状态、退货核销口径、前端展示策略、后续可扩展状态。
- 为客服、门店核销员、产品和研发提供一致解释。

## 2. 当前系统已实现状态（真实生效）

### 2.1 主状态枚举
- `pending`：待核销（用户已下单，等待到店核销）
- `verified`：已核销（门店核销完成，订单履约完成）
- `cancelled`：已取消（含超时自动取消或人工取消）

### 2.2 关键时间字段
- `createdAt`：下单时间
- `timeoutAt`：超时取消时间（用于判断是否自动取消）
- `verifiedAt`：核销时间（仅已核销订单存在）

### 2.3 单号与核销码规则
- `showNo`：预订单业务单号，当前前缀为 `PO`，例如 `PO202604230001`
- `returnNo`：退货申请业务单号，当前前缀为 `RO`，例如 `RO202604230001`
- `verifyCode`：
  - 预订单场景表示“取货码 / 核销码”
  - 退货申请场景表示“退货码”
- 核销台支持直接输入：
  - 预订单单号 `PO...`
  - 退货申请单号 `RO...`
  - 预订单核销码 `verifyCode`
  - 退货申请核销码 `verifyCode`
- 页面展示要求：
  - 预订单分支必须显示“确认核销出库”
  - 退货申请分支必须显示“确认退货回库”
  - 禁止把退货码文案错误展示成“取货码”

### 2.4 状态报告字段（后端统一输出）
- `statusReport.scenario`：`pending | timeout_soon | verified | cancelled | timeout_cancelled`
- `statusReport.cancelReason`：`timeout | manual | null`
- `statusReport.timeoutReached`：是否已达到超时点
- `statusReport.timeoutSoon`：是否处于临近超时窗口（当前为 2 小时）
- `businessStatus`：商家特殊状态（用于在不改变主状态的前提下，向用户同步特殊进度）

### 2.5 商家特殊状态（管理端可手动选择）
- `awaiting_shipment`：待发货（等待接单）
- `preparing`：备货中（已接单）
- `ready`：请取货（等待取货）
- `completed`：已完结（交易结束）
- `after_sale`：其他（售后）
- 当前管理端常规下拉只开放以上 5 项，便于门店按自提业务统一对外口径。
- `shipped`：已发货
- `partially_shipped`：部分发货
- `closed`：已关闭
- `after_sale_done`：售后完成
- `verifying`：核销中
- `verify_failed`：核销失败
- 设计原则：
  - 商家特殊状态只用于补充说明业务进度，不替代主状态 `pending / verified / cancelled`
  - 管理端可在订单查询页详情区手动修改
  - 客户端订单列表与详情页会优先展示这条特殊状态说明，帮助用户及时知晓异常或特殊进展

### 2.6 退货申请状态枚举
- `pending`：待退货核销
  - 表示用户已提交退货申请，门店尚未完成线下确认与库存处理。
  - 核销台查询到该状态时，按钮文案应为“确认退货回库”。
- `verified`：已完成回库
  - 表示门店已完成退货申请核销，对应库存已按原订单状态执行释放或回补。
  - 核销台查询到该状态时，只可查看详情，不可再次核销。
- `rejected`：已拒绝
  - 表示门店已明确拒绝本次退货申请，并记录拒绝原因、处理时间与处理人。
  - 核销台查询到该状态时，只展示结果信息与拒绝原因，不再出现“确认退货回库”按钮。

### 2.6.1 客户端改单与退货触发时机
- `pending`（待取货/未核销）：
  - 客户端不展示“退货服务”与“退货记录”。
  - 客户端提供“修改订单”与“撤回订单”入口。
  - “修改订单”支持在订单详情页内直接调整商品、数量和备注，单笔订单最多成功修改 3 次。
- `verified`（已核销）：
  - 客户端展示“退货服务”与“退货记录”。
  - 仅已核销订单允许客户端发起退货申请。
- 历史兼容口径：
  - 即使旧数据中存在“待取货阶段创建的退货申请”，客户端在 `pending` 阶段也统一不展示，仅保留后台核销与查询能力。

### 2.7 客户端“订单进度”展示规则
- `pending`：
  - 常规显示：`已下单 -> 备货中 -> 待核销 -> 核销后完成订单`
  - 若距离 `timeoutAt` 小于等于 2 小时：显示 `待取货（即将超时）` 和 `请尽快到店核销`
- `verified`：
  - 显示：`已下单 -> 备货完成 -> 已核销 -> 订单完成`
- `cancelled`：
  - 若 `timeoutAt <= 当前时间`：按 `超时自动取消` 展示
  - 否则按 `人工取消` 展示
  - 进度显示：`已下单 -> 备货中 -> 超时自动取消/人工取消 -> 订单关闭`

### 2.8 商家特殊状态与商家留言协同口径
- `businessStatus` 与 `merchantMessage` 职责分离：
  - `businessStatus` 用于结构化分类（例如 `preparing`、`verify_failed`），便于列表筛选与状态统计。
  - `merchantMessage` 用于文本化补充说明，承载“为什么这样处理、下一步怎么做”等上下文。
- 管理端维护规则：
  - 管理端订单工作台支持设置、修改、清空商家留言。
  - 留言输入会执行统一约束：去首尾空格、空字符串按清空处理、长度上限 500 字。
- 客户端展示规则：
  - 订单详情存在有效留言时展示“商家留言”模块。
  - 订单详情留言为空或未设置时，不渲染该模块，避免空占位。
- 一致性要求：
  - 管理端订单详情、管理端订单列表、客户端订单详情均返回同一字段 `merchantMessage`。
  - 空值语义统一为 `null`，避免多端出现 `''/null` 混用导致判断分叉。

## 3. 核销相关状态解释
- 待核销：用户持核销码（二维码/文本码）到店，等待门店执行核销。
- 已核销：核销成功后扣减库存并完成履约。
- 已取消：
  - 自动取消：超过有效领取时段系统关闭订单并释放库存。
  - 人工取消：人工操作关闭订单（如异常订单处理）。

### 3.1 预订单取货核销
- 识别对象：`PO...` 业务单号或预订单 `verifyCode`
- 核销前提：
  - 订单主状态必须为 `pending`
  - 不存在待处理退货申请
- 核销结果：
  - 扣减现货库存
  - 扣减预订库存
  - 订单主状态变为 `verified`

### 3.2 退货申请回库核销
- 识别对象：`RO...` 业务单号或退货申请 `verifyCode`
- 核销前提：
  - 退货申请状态必须为 `pending`
  - 原订单状态与退货申请创建时的来源状态口径一致
- 核销结果分两类：
  - 原订单来源状态为 `pending`：释放对应预订库存
  - 原订单来源状态为 `verified`：回补对应现货库存
- 核销完成后：
  - 退货申请状态变为 `verified`
  - 若原订单为待核销且退货后剩余件数为 0，原订单会转为人工取消
- 页面文案要求：
  - 成功提示应表达“退货核销完成，库存已同步回补”或同等业务含义
  - 禁止沿用“核销出库完成”这类仅适用于取货场景的提示

### 3.2.1 退货申请拒绝
- 触发前提：
  - 退货申请状态必须为 `pending`
  - 管理端工作人员必须填写非空拒绝原因
- 处理结果：
  - 退货申请状态更新为 `rejected`
  - 记录 `rejectedReason`、`handledAt`、`handledBy`
  - 核销台再次查询时只展示结果信息，不再允许继续执行回库核销
- 页面要求：
  - 核销台必须展示“拒绝退货”入口
  - 未填写拒绝原因时，前后端都必须阻止提交并提示“请填写拒绝原因”

### 3.3 待取货订单修改
- 触发前提：
  - 订单主状态必须为 `pending`
  - 必须是订单本人操作
  - 订单不存在退货申请记录
  - 单笔订单累计成功修改次数不得超过 3 次
- 支持内容：
  - 调整原有商品数量
  - 删除原有商品
  - 追加当前仍在售且有库存的商品
  - 修改订单备注
- 库存规则：
  - 数量增加：追加占用对应 `preOrderedStock`
  - 数量减少或删除商品：释放对应 `preOrderedStock`
  - 不直接改动 `currentStock`，因为待取货阶段尚未实际出库
- 页面要求：
  - 修改成功后刷新订单详情、商品明细、总件数与备注
  - 客户端需展示已用次数与剩余次数，次数耗尽后禁用改单入口
  - 取货码继续沿用原订单核销码，不重新生成

### 3.3.1 门店现场改单
- 触发前提：
  - 订单主状态必须为 `pending`
  - 仅管理端工作人员可操作
  - 当前订单不存在退货申请记录
- 支持内容：
  - 增加当前仍启用且已上架到 O2O 商城的商品
  - 减少原有商品数量
  - 删除原有商品
  - 修改订单备注
- 库存规则：
  - 追加商品或增加数量：同步增加对应 `preOrderedStock`
  - 减少数量或删除商品：同步释放对应 `preOrderedStock`
  - 不直接改动 `currentStock`，因为线下实际出库仍发生在核销时
- 页面要求：
  - 核销台待核销预订单详情必须展示“现场改单”入口
  - 保存后，商品明细、总件数、订单备注与后续核销依据都以最新结果为准
  - 订单至少保留一件商品，不允许直接保存空订单

## 4. 订单查询页规则
- 支持按分类筛选：`全部 / 待提货 / 已核销 / 已取消`
- 支持关键词检索：订单号与核销码（可与分类组合）
- 当筛选或关键词无匹配结果时，展示空态提示并保留刷新入口
- 管理端查询与详情接口会先执行一次超时回收，确保同一订单在列表、详情、核销台的状态一致
- 核销台查询规则补充：
  - 输入 `PO...` 或 `RO...` 时，应走按业务单号查询分支
  - 输入核销码 / 退货码时，应走按 `verifyCode` 查询分支
  - 查询结果必须显式标明当前是“预订单”还是“退货申请”

## 5. 行业常见状态（用于后续扩展参考）
> 以下为行业常见状态，不代表本项目当前全部已落地。

- `待付款`：提交订单但未支付
- `待接单/待确认`：商家待确认接单
- `待发货`：已支付，待仓配处理
- `待收货`：已发货，等待签收
- `部分发货`：部分商品已发，剩余待发
- `已完成`：交易完成（通常在签收/核销完成后）
- `已关闭`：超时未支付或关闭订单
- `退款中/售后中`：进入退款或售后流程
- `退款成功/售后完成`：退款或售后处理完成
- `核销中`：适用于线下扫码处理中间态（可选扩展）
- `核销失败`：核销异常（码失效、重复核销、门店权限不足等）

## 6. 推荐状态扩展优先级
- 第一优先级：
  - 增加 `cancelReason`（超时/人工/售后关闭），替代前端推断。
  - 增加 `statusChangedAt`，准确展示每一步状态时间。
- 第二优先级：
  - 增加 `verifying`（核销中）与 `verifyFailed`（核销失败）用于门店高并发扫码场景。
  - 增加 `afterSale*` 子状态，支持退款/售后链路可视化。
