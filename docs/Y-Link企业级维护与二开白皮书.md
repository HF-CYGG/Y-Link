## 1. 系统整体架构总览
Y-Link 是一套面向“线上预订（O2O） -> 线下核销 -> 供货入库”全业务闭环的现代化系统。

### 1.1 核心技术栈
*   **前端基建**：`Vue 3` (Composition API) + `TypeScript` + `Vite`
*   **状态与路由**：`Pinia` (含持久化) + `Vue Router` (严格按需加载与预热)
*   **UI 规范**：`Tailwind CSS` (底层原子约束) + `Element Plus` (业务组件基座)
*   **后端基建**：`Node.js` + `Express` + `TypeORM`
*   **数据库**：支持 `SQLite`（默认一体化/本地）与 `MySQL`（云端分布式）无缝切换。

### 1.2 模块化边界约束
为了保证系统可维护性，所有二次开发必须遵守以下模块边界：
1.  **共享平台层 (`src/components/common`)**：
    *   严禁在此编写任何具体业务逻辑（如直接调用某个订单接口）。
    *   必须保持纯粹的 UI 驱动，通过 `props` 和 `emits` 与业务层通信（例如：`TabbedWorkbenchPage.vue` 工作台基座）。
2.  **业务逻辑层 (`src/composables` & `backend/src/services`)**：
    *   **前端**：复杂的表单联动（如出库开单）必须抽离到 `useXxxForm` 中，Vue 组件仅负责视图绑定。
    *   **后端**：所有核心数据变更必须包裹在 `AppDataSource.transaction` 中，且 `Service` 必须纯粹，不接触任何 `req/res` 对象。
3.  **治理基线层（文档约束）**：
    *   若涉及共享层职责、历史模式禁令、整改范围边界判断，统一以 [Task1 治理基线文档](../spec/task1-全栈治理基线、模块边界、历史模式禁令与范围边界.md) 为准。
    *   若后续任务需要突破本文边界，必须先更新治理文档，再推进代码实现，避免口径漂移。

## 2. 核心业务链路深度解析

### 2.1 O2O 预订与库存状态机
系统采用 **“预扣库存”** 机制防止超卖：
*   **下单瞬间**：客户端下单时，扣减 `preOrderedStock`（预订锁定库存），不影响 `currentStock`（物理实物库存）。
*   **核销动作**：管理端/核销台扫码确认后，同时扣减 `currentStock` 与 `preOrderedStock`，完成状态闭环。
*   **超时释放**：后端依赖系统配置项 `o2o.auto_cancel_hours`，通过自动化脚本或定时任务扫描，将超时的预订单置为 `timeout` 并归还 `preOrderedStock`。

### 2.2 供货方与管理端权限隔离
*   供货方账号 (`supplier` 角色) **绝对禁止** 访问常规业务操作页面。
*   其入口被严格限制在 `supplier-delivery`（送货录入）与 `supplier-history`（历史单据）。
*   前端通过 `routes.ts` 的 `allowedRoles` 结合后端的 JWT 角色校验进行双重防线拦截。

## 3. 日常运维与灾备应急手册 (SOP)

### 3.1 常见故障排查路径
1.  **前端出现“白屏”或“闪烁”**：
    *   *排查点*：检查是否因为后端抛出 `401 Unauthorized` 导致了软跳转残留。
    *   *恢复方案*：前端已封装 `redirectToClientLogin()` 硬跳转，若是新页面，请确保使用该工具而非 `router.push` 退登。
2.  **客户端无法扫码核销**：
    *   *排查点*：`html5-qrcode` 强制要求安全上下文（HTTPS）。
    *   *恢复方案*：本地联调检查是否启动了自签名 HTTPS，线上部署检查 Nginx SSL 证书是否过期。
3.  **Docker 部署下 SQLite 丢失数据**：
    *   *排查点*：检查 1Panel 或 Compose 中 `/app/data` 是否正确挂载到宿主机物理目录。未挂载会导致容器重启数据清空。

### 3.2 数据库平滑迁移 (SQLite -> MySQL)
随着业务量增长，需要从单节点 SQLite 迁移至 MySQL。运维默认流程不再推荐手工导出、手工改环境变量、手工导入数据，而是优先使用管理端的`数据库迁移向导`完成全链路迁移。

推荐 SOP：
1.  先在宿主机侧对 SQLite 文件做一次物理备份，确认备份文件可访问、可恢复。
2.  准备目标 MySQL 空库、账号和最小必要权限，避免直接连接生产共享库。
3.  使用具备`db_migration:view`与`db_migration:operate`权限、且满足管理员门禁的账号登录管理端，进入`系统管理 -> 数据库迁移助手`。
4.  先执行预检，重点确认：SQLite 文件存在、目标 MySQL 可连接、目标库无高风险业务数据冲突、当前运行时覆盖状态可回退。
5.  预检通过后创建迁移任务，并保持`初始化目标表结构`、`执行前创建 SQLite 物理备份`等安全开关开启。
6.  待迁移任务成功后，通过向导写入 MySQL 运行时覆盖配置，并在约定窗口重启后端服务，使新配置正式生效。
7.  切换完成后立即执行关键链路验收；如发现异常，优先通过向导执行`回退到 SQLite`，再安排进一步排障。

适用说明：
*   **优先使用向导**：适用于“已有 SQLite 业务数据，需要平滑迁移到 MySQL”的正式运维场景。
*   **保留手工方式**：仅适用于全新空库初始化、离线环境演练，或由资深运维按专门变更单接管的高级场景。
*   **禁止直接跳过预检**：目标库已有业务数据、连接不稳定、未保留 SQLite 备份时，不应执行正式迁移。

## 4. 代码规范与安全红线 (开发者必读)

### 4.1 注释与文档红线
*   **文件头声明**：所有新增 `.ts` 与 `.vue` 文件顶部必须包含三段式中文注释：`文件职责`、`实现逻辑`、`维护说明`。
*   **禁止机翻**：严禁使用类似 `// Fetch data` -> `// 获取数据` 这种毫无意义的注释。必须写明**业务上下文**（如：`// 获取数据：此处为了防止闪烁，必须使用 useStableRequest 包裹`）。

### 4.2 安全与稳定性红线
*   **频控校验**：所有面向客户端的变更类接口（登录、改密、注册）必须在 `authSecurityService.ts` 注册限流规则。
*   **防止竞态**：前端高频筛选列表，严禁直接 `await axios`，必须使用 `runLatest` 调度，否则会导致旧请求覆盖新结果。
*   **防 NaN 污染**：任何从外部/数据库解析的数字（如金额、次数），必须进行 `Number.isFinite` 校验与默认值兜底，避免 `NaN` 导致整个页面崩溃。
