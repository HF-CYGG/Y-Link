# Y-Link 生产大表低风险结构与数据迁移方案 (以 base_product 为例)

## 背景
业务模块对 `base_product` 表新增了 O2O 相关列（如：`current_stock`、`o2o_status` 等），且定义为 `NOT NULL DEFAULT ...`。
在大表（百万行级以上）、长事务环境或老版本 MySQL 中，直接执行该 DDL 可能引发：
- **长时间的元数据锁等待（MDL Wait）**：导致全站响应堆积卡死。
- **全表重写与复制延迟**：不仅占用大量磁盘 IO，还会使得主从同步延迟飙升。
- **业务事故（更致命）**：如果直接使用默认的 0 库存和未上架状态回写，历史商品可能瞬间“被售罄下架”。

为避免此类情况，本方案采用“分阶段 + 批处理”策略。

---

## 一、上线窗口与操作步骤

### 阶段一：加列与预演（提前数天/低峰期执行）
> **目标**：不影响业务，只修改表字典定义，将新列安全加上。
1. 检查 MySQL 中的长事务（`SHOW PROCESSLIST;` 或 `SELECT * FROM information_schema.innodb_trx;`），排空正在操作 `base_product` 表的卡顿请求。
2. 连接生产主库，执行第一步拆分的 SQL（允许 NULL，不设默认值）。
   - **脚本**：`backend/sql/migrations/step1_add_nullable_columns.sql`
   - **效果**：数据库瞬时完成元数据修改。因为没带默认值也不强求非空，因此不会触发表内旧数据的大量回填和重写。

### 阶段二：脚本分批订正老数据（提前一至两天/业务平峰期）
> **目标**：通过业务脚本控制订正节奏，将历史数据的 NULL 值刷为业务期望的值（如 0 或真实对接的库存）。
1. 登录一台与数据库同网段的应用服务器（或堡垒机）。
2. 执行回填订正脚本。
   - **脚本**：`backend/scripts/step2-backfill-data.ts`
   - **效果**：该脚本使用“基于主键范围的游标扫描 + 小步快跑 (批量 500 行) + 每批休眠 50ms”的机制，极大地降低了写压力，不会堵死其他在线 DML 操作。
3. 如果你们的历史库存是从另外系统对接进来的，**务必在这个阶段把脚本里的默认 0 替换为真实的库存拉取逻辑**。

### 阶段三：应用发布与约束收紧（发布窗口执行）
> **目标**：发布包含新字段实体定义的前后端代码，并在确认一切平稳后，锁死表结构（NOT NULL）。
1. 发布新版 Y-Link 后端与前端。
2. 观察 30 分钟，确认新下单/商品详情等无报错。
3. 执行最后一步 SQL（加严约束）。
   - **脚本**：`backend/sql/migrations/step3_tighten_constraints.sql`
   - **效果**：此时老数据已经在上一步订正完毕，表内没有 NULL 值，该 DDL 执行时只需做校验和字典更新，同样也是极快的。

## 二、回滚预案

在迁移的任何一步发现问题，都可以执行对应的回滚措施。

### 情景 1：执行 Step 1 时出现严重的 MDL 锁表
- **现象**：数据库报警，大量请求出现锁等待超时。
- **原因**：现场有未发现的僵尸长事务死死占住了旧表的元数据锁，导致 DDL 被阻塞，后续请求又被该 DDL 阻塞。
- **预案**：
  1. 立刻通过 `KILL <thread_id>` 杀掉该 `ALTER TABLE` 语句。
  2. DDL 回滚极快，杀掉后锁释放，业务请求立刻恢复。
  3. 排查清楚到底是谁在锁表，清理干净后重新挑窗口执行。

### 情景 2：执行 Step 2 (数据回填) 期间主从延迟过大或 CPU 飙升
- **现象**：从库同步跟不上，导致读库数据陈旧。
- **预案**：
  1. `Ctrl+C` 直接中断 `step2-backfill-data.ts` 进程。
  2. 因为是分批小事务，中断后没有任何问题。
  3. 调大脚本里的休眠时间（如 `50ms` -> `200ms`）或调小 `BATCH_SIZE`（`500` -> `100`）后，重新运行脚本即可，它会自动断点续传。

### 情景 3：应用发布后发现 O2O 逻辑有严重 Bug，需回退代码
- **现象**：新版代码引发阻断性故障，必须回滚到旧版本应用镜像。
- **预案**：
  1. 直接回滚应用镜像和代码。
  2. **表结构无需回滚**（因为旧版代码完全不认识这些新加的 `o2o_status`、`current_stock` 列，在 ORM 映射时会自动忽略，不会引发报错）。
  3. 即使你已经执行到了 Step 3 (NOT NULL DEFAULT)，由于旧版代码写入时带了 DEFAULT 占位，也不至于发生插入拦截。待问题修复后重新发布即可。
