# Y-Link 客户端 H5 企业级 UI/UX 与功能架构优化方案

> 文档说明：本文是客户端体验优化与架构升级方案，用于统一设计方向、组件拆分原则和性能治理要求。
> 适用范围：客户端 H5 的页面重构、视觉升级、交互设计、状态管理与性能优化评审。
> 维护口径：本文包含“已落地方向 + 规划目标”两类内容；若某项能力尚未开发完成，实施时必须以代码现状和任务排期为准，不得把规划描述误写成已上线事实。

## 1. 顶层设计理念与全局 UI 规范
深度参考业界顶尖 O2O 预订与点单系统（如瑞幸咖啡、喜茶 GO）以及优秀开源电商基建（如 Litemall、Mall4j、Vant UI），客户端需彻底摒弃传统“后台表单式”呈现，转向 **B2C 消费级** 体验。核心贯彻 **Bauhaus 极简风格** 与 **沉浸式交互**。

### 1.1 视觉与材质规范 (Visual & Material)
* **沉浸式与无界感 (Borderless)**：取消传统的顶部生硬导航栏 (NavBar)，采用沉浸式头部（背景渐变或图片延伸至状态栏），结合页面向下滚动时的毛玻璃吸顶效果。
* **触控与无障碍优先 (Touch-First & A11y)**：所有高频交互元素最小触控区域 `44x44px`。文字对比度符合 WCAG 标准，确保户外强光下依然清晰可见。
* **Design Token 设计令牌系统**：参考 Vant/NutUI 等开源组件库，建立基于 CSS 变量的全局 Theme 体系（如 `--ylink-color-primary`、`--ylink-radius-lg`），剥离硬编码，为未来可能的主题切换（如暗黑模式或节日皮肤）打好基础。
* **材质层级 (Z-Index)**：
  * **底层**：`#F5F7FA` 极浅灰/米白，提供呼吸感。
  * **内容层**：纯白卡片，全圆角 (`16px` 或 `20px`)，无边框，仅用弥散阴影 (`0 8px 24px rgba(0,0,0,0.04)`) 区分层级。
  * **悬浮层**：底部结算栏、弹窗使用 **毛玻璃效果 (Backdrop Blur)** 确保视线连贯性。

### 1.2 开源项目借鉴结论与落地映射
结合开源项目的公开说明，可以提炼出一组适合 Y-Link 的高复用设计规律：
* **Litemall 的启发**：其用户端功能覆盖首页、分类、搜索、商品详情、购物车、下单、订单列表、订单详情、收藏与意见反馈，说明成熟商城普遍采用“首页引流 + 分类承接 + 详情转化 + 购物车汇总 + 订单闭环”的链路组织方式，Y-Link 客户端也应按此完整链路来规划，而不是只做“商品列表 + 提交订单”的简化页面。
* **Mall4j 的启发**：其强调完整 SKU 与下单流程、多端呈现、生产环境可维护性，说明商品售卖页面不能只关注展示，还要提前规划规格扩展、订单状态机、异常兜底与多端一致体验。
* **Store-Web-App 的启发**：其将首页、分类页、购物车页、个人页拆分为稳定主 Tab，并通过懒加载、状态管理、滚动优化提升体验，说明客户端主体结构应当固定成“商城 / 订单 / 我的”三段式，不建议把所有入口堆叠在一个超长页面中。
* **Vue3 + Vant 购物车实践的启发**：购物车状态持久化、SKU 联动、防重复提交、弱网容错被视为高转化商城的基础能力，因此 Y-Link 文档需要明确把“状态持久化、并发锁、接口幂等、库存最终以后端为准”写成强制规范。

### 1.3 Y-Link 场景化取舍原则
由于本项目以“校内预订、线下自提、后台核销”为核心，不必照搬大型综合商城的复杂营销体系，设计上应坚持以下取舍：
* **保留高频核心能力**：分类、搜索、商品详情、购物车、确认订单、订单详情、核销码、个人中心。
* **弱化低价值复杂度**：直播、秒杀、分销、复杂优惠券、积分商城等暂不纳入客户端主方案。
* **突出线下履约特征**：所有页面围绕“看商品 -> 预订 -> 等待备货 -> 现场扫码取货”设计，不把页面重心放在支付成功页，而是放在“待提货订单”和“核销码展示页”。

## 2. 核心功能页面重构与组件拆分

将目前冗余的单文件页面打散，全面对标开源电商与外卖体系：

### 2.1 首页/商城页 (双栏联动经典布局)
借鉴“外卖点单”经典布局，实现一站式选购：
* **路由状态缓存 (`<KeepAlive>`)**：利用 Vue 的 KeepAlive 缓存整个商城页面，确保用户从“订单页”或“我的页”切回时，左侧分类高亮和右侧滚动位置精准保留。
* **头部信息带设计**：参考成熟商城的首页组织方式，顶部建议固定为“品牌标题 + 搜索框 + 公告条/活动说明”三段结构。即使没有营销活动，也可以承载“营业时间、提货须知、库存提醒、公告通知”等实用信息。
* **首页内容分层**：商城页不应只有分类与商品列表，还应在列表上方保留一层轻量运营区，包括轮播 Banner、推荐分组、近期热销或新到商品入口，用于提升首页的信息密度与视觉完成度。
* **双栏联动核心区 (`CategorySidebar` + `ProductList`)**：
  * **左侧**：一级分类侧边栏，点击时右侧平滑滚动定位（Scroll Into View）。
  * **右侧**：商品瀑布流/列表，支持分组吸顶（Sticky Header）。
* **商品卡片 (`ProductCard`)**：图片占比较大；包含标签槽（“热销”、“仅剩3件”）；右下角集成高亮加购步进器。
* **商品卡片信息层级**：名称优先，其次为规格摘要、库存提示、预订数量、价格或积分说明。若当前项目无需复杂价格对比，也应保留“当前可预订 / 已被预订”双指标展示，以体现库存实时感。
* **搜索结果单独态**：搜索命中后切换至独立结果态，弱化左侧分类，避免用户在搜索结果和分类联动之间来回跳动。

### 2.2 半屏抽屉式详情与 SPU/SKU 引擎 (Bottom Sheet)
* 摒弃传统的整页跳转查看详情。点击商品卡片时，从底部弹出**大半屏抽屉 (Drawer)** 展示高清大图和详细描述。
* **SPU/SKU 规格矩阵计算**：若未来涉及商品多规格（如颜色、尺码），底层抽屉需集成开源电商标准的 SKU 选择算法（如基于图的邻接矩阵路径算法），实现无效规格的自动置灰（Disable）与价格实时联动。
* **详情抽屉信息结构**：建议固定为“图片 / 标题 / 简介 / 库存状态 / 规格区 / 数量步进器 / 底部主按钮”。减少滚屏层级，让用户在一个视口内完成浏览与加购。
* **相关推荐与替代品**：当库存不足或商品停售时，详情抽屉底部可追加“相似推荐 / 可替代商品”，该做法在多个开源商城中都用于减少流失。

### 2.3 沉浸式确认订单页 (Checkout)
* 采用卡片分组显示：【提货人信息】、【提货时间/备注】、【商品明细】。
* 底部固定强引导的“确认预订”栏。
* **确认页校验策略**：在提交前完成手机号、部门名称、限购规则、库存变化、备注字数等前端即时校验；真正提交时仍以后端二次校验为准，并在返回结果中明确提示哪一项失败，避免笼统报错。
* **价格与数量汇总吸底**：借鉴成熟购物车与结算页设计，底部吸底栏应同时展示“商品件数、总价/总积分、预计提货时间、提交按钮”。

### 2.4 核销凭证与订单详情页 (Ticket & QR Code)
* **取货凭证高亮**：进入“待提货”订单详情时，自动将屏幕亮度调高（调用 Web API，若支持）。
* **大号核销码**：页面中心展示巨大、高对比度的二维码/条形码，上方标注大字号“取货码：XXXX”。
* **状态机与时间轴 (State Machine & Timeline)**：底部以时间轴形式清晰展示订单流转（已下单 -> 备货中 -> 已核销 -> 已评价）。
* **自提指引区**：参考票券型页面设计，在核销码下方增加“取货地址、取货时段、联系人、温馨提醒”模块，减少用户到店后的沟通成本。
* **异常订单视图**：已取消、已超时、部分缺货、已核销等状态应提供差异化视觉稿，而不是只改一行文字。

### 2.5 购物车页与底部结算层
* **购物车页角色定位**：借鉴开源商城的标准结构，购物车页负责“二次确认与批量调整”，不承担商品发现任务。
* **核心交互**：支持全选、单选、批量删除、数量步进、库存失效提示、失效商品单独分组。
* **空购物车设计**：必须提供精致空状态页，附带“去逛逛”按钮，避免空白页面造成产品未完成感。
* **迷你购物车抽屉**：在商城页底部结算栏点击后弹出购物车抽屉，适合快速调整数量；需要更完整操作时再进入独立购物车页。

### 2.6 订单列表页与个人中心页
* **订单列表页**：建议拆成“全部 / 待提货 / 已完成 / 已取消”标签页，首屏默认进入“待提货”，因为这与线下履约最相关。
* **订单卡片结构**：商品摘要、订单编号、下单时间、状态标签、金额/件数、主要操作按钮（查看详情、再次下单）。
* **个人中心页**：采用轻量结构，不堆砌复杂资产模块，保留账号信息、常用联系人、收货/提货信息、反馈建议、退出登录即可。

## 3. 动效与微交互设计 (Micro-interactions)

参考 Material Design 与主流移动端框架，通过 CSS/Vue Transition 注入“灵魂”：

* **抛物线飞入购物车 (Parabolic Animation)**：点击“+”号加购时，商品缩略图化为一个圆点，以抛物线轨迹飞入底部购物车图标，提供强烈的“获得感”。
* **Vue 3 自定义指令赋能**：沉淀业务指令来降低模板复杂度，例如：
  * `v-ripple`：Material Design 风格的水波纹触控反馈。
  * `v-debounce`：防止连点的全局防抖指令。
* **购物车气泡弹跳 (Bounce)**：数字增加时，底部红点角标执行弹跳与放大缩小动画。
* **骨架屏先于加载图 (Skeleton Screen)**：摒弃全局菊花图 (Spinner)。列表加载时，展示对应结构的灰色闪烁骨架块，平滑过渡至真实数据，降低等待焦虑。
* **路由推拉转场 (iOS-like Transitions)**：页面间前进/后退使用 `slide-left` / `slide-right`。
* **空状态与异常状态动画**：空购物车、无搜索结果、网络异常、订单为空时，统一使用轻量插画或骨架转场，避免突兀的纯文本提示。

## 4. 性能优化与前端架构 (Performance & State)

吸收优秀开源架构的底层性能优化手段：

* **长列表虚拟滚动 (Virtual Scrolling)**：若商品分类与数量超 100 个，右侧列表引入虚拟列表技术（如 `vueuse/useVirtualList`），只渲染可视区域 DOM，彻底解决低端机型滚动卡顿。
* **接口聚合与预加载 (Prefetching)**：利用 Vue Router 的钩子，在进入商城页前或应用空闲时，预加载关键数据与图片（WebP格式 + `v-lazy`），实现“秒开”体验。
* **全局错误降级与断网重试**：封装统一的请求拦截器，在遭遇 5xx 错误或无网络时，平滑切换至精美的缺省页（Empty State），并提供一键重试机制，避免白屏。
* **状态精细化与持久化管理 (Pinia)**：全局维护 `client-cart` 模块，本地缓存 (LocalStorage) 购物车快照，防止页面意外刷新导致用户选购丢失。
* **Store 分层建议**：至少拆分为 `client-auth`、`client-cart`、`client-catalog`、`client-order` 四类 Store，避免一个 Store 同时承载身份、商品、购物车、订单全部状态。
* **服务端权威原则**：借鉴成熟商城实践，前端展示库存与价格仅用于即时反馈，提交订单时必须以后端实时校验结果为准；文档中要明确“前端可乐观展示，后端做最终裁决”。
* **缓存失效策略**：分类、商品列表、个人偏好、购物车快照分别设置不同生命周期，避免一刷新就全量重拉，也避免旧库存长期污染页面。
* **组件复用策略**：把商品卡片、价格区、状态徽标、空态页、骨架屏、异常提示条做成可复用组件，减少后续订单页和商城页出现两套风格。

## 5. 实施路线图 (Phased Implementation)

为保证业务稳定性，建议分三个阶段逐步重构：

### 阶段一：骨架与双栏布局 (基础重构)
1. 建立 Design Token 变量体系，规范全局色值、圆角与阴影。
2. 引入并封装 `ClientLayout`（顶部沉浸 + 底部毛玻璃 TabBar）及 `<KeepAlive>` 路由缓存。
3. 实现首页“头部信息带 + 左侧菜单 + 右侧列表”的双栏联动滚动逻辑。
4. 补齐空购物车、无订单、无搜索结果三类基础空状态页面。

### 阶段二：组件打磨与微交互 (体验升华)
1. 重构 `ProductCard`，封装独立的步进器与半屏详情抽屉（预留 SPU/SKU 扩展口）。
2. 引入 Pinia `client-cart` 接管购物车，实现底部悬浮结算栏与数据持久化。
3. 开发“抛物线飞入购物车”动效与骨架屏 (Skeleton)，落地 `v-debounce` 自定义指令。
4. 完成购物车抽屉、独立购物车页、确认订单页三段式下单链路。

### 阶段三：核心链路闭环与性能 (高阶优化)
1. 重构“确认订单页”与“订单详情页”，加入大号高亮核销二维码设计与状态时间轴。
2. 注入全局路由左右推拉动画 (`slide-left`/`slide-right`) 与断网缺省页处理。
3. 接入长列表虚拟滚动与接口预加载机制，完成移动端真机多尺寸视口兼容测试。
4. 补齐性能验证、弱网回归、订单提交流程回归与核销链路验收。
