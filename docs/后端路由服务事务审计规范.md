# Y-Link 后端路由服务事务审计规范

> 文档更新时间：2026-05-01
> 适用对象：后端开发、接口联调、系统治理维护人员

## 1. 文档目的
- 本文档用于统一 Y-Link 后端路由、服务、事务、权限与审计的写法，避免继续出现“路由层漏门禁、服务层直接碰 req/res、写操作无审计、事务边界不清”的历史模式。

## 2. 目录职责约定
- `backend/src/routes`
  - 负责鉴权、中间件拼装、参数校验、HTTP 返回结构。
  - 不承载业务计算与跨表写逻辑。
- `backend/src/services`
  - 负责业务规则、事务边界、跨仓储协调、审计入参整理。
  - 不接触 `req/res` 原始对象。
- `backend/src/entities`
  - 只定义实体结构、字段映射、约束注释。
- `backend/src/middleware`
  - 负责登录态、角色、权限等横切逻辑。
- `backend/sql`
  - 负责数据库增量迁移和数据库级兜底约束。

## 3. 路由层规范
- 路由入参必须使用 schema 校验，禁止直接信任前端提交内容。
- 高危接口必须显式挂接门禁：
  - `requirePermission(...)`
  - 必要时叠加 `requireRole(...)`
- 路由层只做这些事：
  - 取当前登录上下文
  - 校验参数
  - 调服务
  - 统一响应
- 路由层禁止：
  - 直接写数据库
  - 直接拼接复杂 SQL
  - 直接承载审计内容拼装细节

## 4. 服务层规范
- 服务层是业务真源：
  - 订单、O2O、数据维护、系统配置等业务规则必须沉淀在 service。
- 服务方法要尽量具备稳定输入输出：
  - 输入对象类型明确
  - 返回 DTO/视图结构明确
- 服务层禁止读取 `req.body`、`req.user` 之类 HTTP 细节，调用方必须先转换成明确的业务输入。

## 5. 事务规范
- 跨表写入、库存变化、状态流转、配置切换等关键写操作必须使用事务。
- 典型必须上事务的场景：
  - 开单主表 + 明细写入
  - O2O 核销与库存扣减/回补
  - 数据导入
  - 数据库迁移任务状态切换
  - 高危配置写入与运行时覆盖落盘
- 事务内要保证：
  - 先校验，再写入
  - 所有关键仓储共用同一 `manager`
  - 失败时不留下半成功脏数据

## 6. 审计规范
- 以下写操作必须有审计：
  - 系统配置修改
  - 数据备份、导入
  - 数据库迁移创建、执行、切换、回退、清理覆盖
  - 用户创建、编辑、启停、重置密码
  - 关键业务状态修改
- 审计至少记录：
  - 操作者
  - 动作类型与动作标签
  - 目标对象类型、ID、业务编码
  - 结果状态
  - 必要的上下文摘要
- 审计内容禁止记录敏感明文：
  - 密码
  - 敏感配置原值
  - 不必要的完整隐私数据

## 7. 权限与角色双门禁规范
- 默认以权限点为主，角色为补充。
- 适合只用权限点的场景：
  - 普通业务增删改查
  - 前后端都已共识的动作级能力
- 必须叠加角色限制的场景：
  - 数据库迁移、运行时切换、系统级灾备能力
  - 仅管理员可触达的治理型高危动作
- 新接口上线前必须回答两个问题：
  - 需要什么权限点
  - 是否还需要角色硬边界

## 8. 数据完整性规范
- 业务字段校验不能只靠前端。
- 服务层必须再次校验：
  - 长度
  - 枚举值
  - 数量/价格/库存边界
  - 跨字段关系
  - 跨表引用存在性
- 对关键数值字段应尽量补数据库级约束，形成双保险。

## 9. 配置治理规范
- 敏感配置读取与写入必须区分：
  - 对外展示返回脱敏值
  - 真正发送/连接时读取真实值
- 运行时覆盖配置写盘要使用原子写策略，避免半写入损坏。
- 配置缺失不能依赖非空断言掩盖，必须抛出明确可读错误。

## 10. 推荐接入模板
- 新路由：
  - schema 校验
  - `requirePermission`
  - 取 `auth` 上下文
  - 调 service
  - 返回统一 JSON
- 新 service 写操作：
  - 输入 normalize
  - 事务包裹
  - 调仓储
  - 写审计
  - 返回前端需要的视图结构

## 11. 回归检查清单
- 路由是否漏挂权限或角色门禁。
- 服务是否绕开事务直接多表写入。
- 写操作是否补齐审计。
- 敏感字段是否被直接回显或写入日志。
- 数据校验是否只在前端存在、后端缺兜底。
