# 按开发规划落地 O2O 预订系统 Spec

## Why
当前已完成《开发规划》文档，但尚未进入可交付实现阶段。需要将规划转为可执行规格，确保客户端预订、管理端出入库核销、库存一致性与可运维能力一次性落地并可自动化验证。

## What Changes
- 新增客户端 H5 预订端能力：注册登录、密码找回、商品大厅、下单、订单详情与核销码展示。
- 新增管理端 O2O 能力：商品大厅维护（上下架/预览图/详情）、预订单核销台、入库管理、业务规则配置页。
- 新增库存双字段业务规则：`current_stock` 与 `pre_ordered_stock` 的一致性计算与事务更新。
- 新增验证码双通道策略：短信验证码优先，未配置短信时自动降级图形验证码。
- 新增可运维能力：SQLite 物理备份、跨库 JSON 导出导入、面向 MySQL 的平滑迁移路径。
- 新增自动化测试与发布前质量门禁，覆盖核心业务链路与并发库存场景。
- **BREAKING**：无。

## Impact
- Affected specs: 客户端认证、商品展示、预订下单、订单核销、库存管理、系统配置、数据备份恢复。
- Affected code: 前端路由与页面模块、后端认证与订单服务、库存事务服务、系统配置模块、数据库实体与迁移、自动化测试脚本。

## ADDED Requirements
### Requirement: 客户端 H5 认证与找回
系统 SHALL 提供客户端用户注册、登录与密码找回能力，并具备防机器人校验与降级机制。

#### Scenario: 短信可用时认证
- **WHEN** 系统已配置短信服务且用户进行注册、登录验证或密码找回
- **THEN** 系统使用短信验证码完成身份校验并允许后续流程

#### Scenario: 短信不可用时降级
- **WHEN** 系统未配置短信服务或短信通道不可用
- **THEN** 系统自动启用图形验证码并结合频率限制完成反机器人防护

### Requirement: 客户端商品大厅与下单可视化
系统 SHALL 在客户端展示可订购商品、实时可用库存与已预订数量，并支持下单后查看订单详情。

#### Scenario: 实时库存展示
- **WHEN** 用户浏览商品大厅
- **THEN** 每个商品展示“当前剩余数量”与“已被预订数量”，且数值基于最新库存计算

#### Scenario: 下单后查看详情
- **WHEN** 用户提交预订单成功
- **THEN** 用户可在订单详情页查看明细、状态与核销二维码

### Requirement: 管理端商品大厅维护
系统 SHALL 在管理端提供商品大厅维护页面，支持上下架、预览图编辑与详情编辑。

#### Scenario: 上下架控制
- **WHEN** 管理员将商品设置为下架
- **THEN** 该商品不在客户端大厅展示且不可被新下单

#### Scenario: 图文信息维护
- **WHEN** 管理员更新商品预览图与详情内容
- **THEN** 客户端商品展示信息同步更新

### Requirement: 管理端出入库与核销闭环
系统 SHALL 由管理端工作人员执行入库与出库核销，形成完整库存闭环。

#### Scenario: 扫码核销出库
- **WHEN** 工作人员在核销台扫码订单并确认出库
- **THEN** 系统更新订单状态并按规则扣减库存字段且记录流水

#### Scenario: 入库登记
- **WHEN** 工作人员提交入库单
- **THEN** 系统增加物理库存并记录入库流水

### Requirement: 可配置业务规则（默认开启）
系统 SHALL 支持在管理端配置“订单超时自动取消”和“限购策略”，并提供合理默认值。

#### Scenario: 默认策略生效
- **WHEN** 系统首次部署或配置重置
- **THEN** 自动开启超时取消与限购策略，默认超时为 24 小时、默认单人单次限购为 5 件

#### Scenario: 管理端关闭或调整策略
- **WHEN** 管理员在系统配置页修改开关或参数
- **THEN** 新规则立即对后续订单生效且可审计

### Requirement: 数据备份恢复与 MySQL 迁移
系统 SHALL 支持 SQLite 阶段备份恢复，并提供数据库无关的数据迁移能力。

#### Scenario: SQLite 物理备份
- **WHEN** 管理员执行全量备份
- **THEN** 系统生成可下载的 SQLite 备份文件

#### Scenario: 跨库迁移
- **WHEN** 管理员从 SQLite 导出并在 MySQL 环境导入
- **THEN** 业务数据可完整恢复且关键关系保持一致

### Requirement: 自动化测试与质量门禁
系统 SHALL 在开发完成后执行自动化测试与性能回归校验，确保核心链路可用。

#### Scenario: 核心自动化测试通过
- **WHEN** 执行项目自动化测试套件
- **THEN** 客户端认证、下单、核销、库存一致性、配置策略与备份恢复相关测试全部通过

#### Scenario: 性能回归校验通过
- **WHEN** 完成核心页面与路由改造后执行性能验证
- **THEN** `npm run verify:performance` 通过且报告无阻断项

## MODIFIED Requirements
### Requirement: 系统配置能力扩展
现有系统配置 SHALL 增加“线上预订设置”分组，包含超时取消开关与时长、全局限购开关与限额参数，并对修改行为保留审计信息。

## REMOVED Requirements
### Requirement: 无
**Reason**: 本次是在现有系统上扩展 O2O 能力，不删除既有能力。  
**Migration**: 无。
