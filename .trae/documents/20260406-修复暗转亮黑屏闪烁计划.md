# 修复暗转亮黑屏闪烁计划

## Summary
本次计划聚焦修复“从暗色切换到亮色时出现一下黑屏闪烁”的问题。当前主题系统已经具备基于点击点的 View Transitions 圆形扩散/收缩能力，因此本次不重做主题系统，而是在现有实现上精确修正暗转亮路径的视图叠层与动画策略，消除黑屏闪烁，同时保留现有的全局主题过渡质感与降级能力。

## Current State Analysis
- 当前主题切换核心逻辑集中在 [theme.ts](file:///f:/Y-Link/src/store/modules/theme.ts#L279-L367)，由 `runViewTransition` 决定 View Transition 的主路径，由 `runFallbackTransition` 负责不支持 API 时的降级路径。
- 现有实现中：
  - 亮转暗时，动画对象是 `::view-transition-new(root)`，即“新暗色视图”做扩散揭示。
  - 暗转亮时，动画对象是 `::view-transition-old(root)`，即“旧暗色视图”做收缩隐藏。
- 当前 View Transition 叠层规则定义在 [style.css](file:///f:/Y-Link/src/style.css#L163-L188)：
  - `data-theme-transition-mode='dark'` 时，新视图在上层。
  - `data-theme-transition-mode='light'` 时，旧视图在上层。
- 结合现有代码结构，暗转亮闪烁的高概率根因是：当前“暗转亮”路径通过裁剪旧暗色截图来露出下层亮色视图，这种做法在浏览器合成层切换期间更容易短暂暴露到底层背景/透明区，视觉上表现为黑屏或黑闪。
- 当前 `ThemeToggle` 本身已升级，且从状态层看没有额外缺陷；问题核心不在按钮组件，而在暗转亮时的视图过渡策略与层级组织。
- 当前降级路径 `fallback` 仅使用受控颜色过渡与点击点光晕提示，[style.css](file:///f:/Y-Link/src/style.css#L94-L161)，与“黑屏闪烁”问题无直接冲突，因此本次只需要验证降级路径不被回归。

## Proposed Changes

### 1. 调整暗转亮的 View Transition 主路径
- 文件：[theme.ts](file:///f:/Y-Link/src/store/modules/theme.ts)
- What
  - 将暗转亮时的动画目标从“裁剪旧视图”调整为“揭示新视图”。
  - 让亮转暗、暗转亮两条路径都统一为“目标主题截图在上层扩散/收缩显现”的策略，而不是一条揭示新图、一条裁剪旧图。
- Why
  - 统一策略更稳定，也更符合用户主观感受：切到什么主题，就让什么主题从点击点涌现出来。
  - 避免旧暗色截图收缩时短暂露出底层黑底/透明层，降低黑闪概率。
- How
  - 保留 `startViewTransition()` 与点击点半径计算逻辑不变。
  - 重写 `runViewTransition` 中 `pseudoElement` 和 `clipPath` 的选择逻辑：
    - 亮转暗：继续以 `::view-transition-new(root)` 做扩散。
    - 暗转亮：改为以 `::view-transition-new(root)` 做扩散，或统一抽象为“总是对新视图做 reveal 动画”。
  - 如需保留“收缩”感，则通过反向关键帧驱动 `new(root)`，而不再裁剪 `old(root)`。

### 2. 重构 View Transition 叠层规则，避免底层背景暴露
- 文件：[style.css](file:///f:/Y-Link/src/style.css)
- What
  - 调整 `data-theme-transition-mode='light'` 的旧/新视图层级关系。
  - 视情况补充 `::view-transition-group(root)` / `::view-transition-image-pair(root)` 的稳定化样式。
- Why
  - 当前暗转亮把旧视图放在上层，本质上是为了配合“旧视图收缩”方案；如果改成“新视图揭示”，叠层规则也需要同步调整，否则会互相抵消。
  - 明确 group / pair 的显示上下文，能减少浏览器合成层在切换瞬间的黑边、透明缝或默认交叉淡入干扰。
- How
  - 保持默认 cross-fade 关闭。
  - 让暗转亮、亮转暗两种模式都以“新视图在上、旧视图在下”为优先策略。
  - 如浏览器实测仍存在瞬间空层，再补充：
    - `::view-transition-group(root) { isolation: isolate; }`
    - 必要的背景色/overflow/contain 稳定化样式。

### 3. 保持降级路径与按钮体验不回归
- 文件：[theme.ts](file:///f:/Y-Link/src/store/modules/theme.ts)、[ThemeToggle.vue](file:///f:/Y-Link/src/layout/components/ThemeToggle.vue)
- What
  - 不改动 ThemeToggle 的视觉设计，只确认交互入口仍调用统一 Store。
  - 不改动 fallback 方案的功能语义，仅在验证阶段确认修复主路径后其行为不回退。
- Why
  - 当前问题是暗转亮黑闪，不是按钮交互退化，也不是无 API 环境失效。
  - 收敛改动面可以降低把一个单点问题扩散成整套主题系统回归的风险。
- How
  - ThemeToggle 保持现有 `toggleTheme(event)` 入口。
  - 修复完成后重点验证：
    - 支持 View Transition 时不黑闪；
    - 手动禁用 `document.startViewTransition` 后 fallback 依旧可用。

### 4. 浏览器级针对性验收
- 文件：无代码新增，验收使用现有本地环境
- What
  - 针对登录页和主系统页进行暗转亮、亮转暗反复切换验证。
- Why
  - 当前 bug 是视觉与合成层问题，只看代码或构建日志不够，必须做浏览器实际回归。
- How
  - 在 `/login` 与 `/dashboard` 上分别验证：
    - 暗 -> 亮不出现黑屏闪烁；
    - 亮 -> 暗仍保持现有效果；
    - 控制台无新增 error/warn；
    - 页面无明显布局抖动。

## Assumptions & Decisions
- 已确认当前本地环境有运行中的前端联调进程，可用于浏览器级验证，不需要额外搭新环境。
- 这次计划默认“不重做主题系统结构”，而是在现有 [theme.ts](file:///f:/Y-Link/src/store/modules/theme.ts) 和 [style.css](file:///f:/Y-Link/src/style.css) 上精修。
- 默认采用“统一以新视图作为主题揭示层”的策略，这是当前最小、最稳且最符合交互直觉的修复方向。
- 如果实测发现仅切换动画目标仍不足以彻底消除黑闪，则第二优先补充 View Transition group/pair 的隔离样式，而不是扩大到更多页面级逻辑修改。
- 当前计划不包含额外视觉升级（如液态边缘、更多粒子效果），只解决暗转亮黑闪问题。
