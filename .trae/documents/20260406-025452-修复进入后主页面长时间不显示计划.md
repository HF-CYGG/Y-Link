# 修复进入后主页面长时间不显示计划

## Summary
目标是修复“本地 `npm run dev` 调试时，登录后首进系统，壳层先出现，但主内容区长时间不显示”的问题。成功标准是：登录成功后尽快看到可感知的主内容占位或真实内容；即使工作台数据仍在加载，也不能出现大块无反馈空白区；现有登录态恢复、权限守卫、路由缓存与性能优化能力不回退。

## Current State Analysis
- 根组件 [App.vue](file:///f:/Y-Link/src/App.vue#L1-L4) 只有裸 `RouterView`，没有顶层异步路由兜底。
- 主系统壳层 [AppLayout.vue](file:///f:/Y-Link/src/layout/AppLayout.vue#L64-L132) 已有 `Suspense` fallback，但它只覆盖壳层内部的子页面，无法覆盖“壳层已出现但子页面未稳定显示”的体验缺口。
- 路由守卫 [index.ts](file:///f:/Y-Link/src/router/index.ts#L40-L92) 会在进入业务页前等待 `initializeAuth()`，而预热逻辑只在 `afterEach` 触发，当前首跳收益有限。
- 登录链路 [auth.ts](file:///f:/Y-Link/src/store/modules/auth.ts#L180-L194) 在登录成功后直接进入过渡态，但没有显式保证“当前目标页面”已预热。
- 工作台页面 [DashboardView.vue](file:///f:/Y-Link/src/views/dashboard/DashboardView.vue#L56-L83) 虽有 `el-skeleton`，但它只包裹“今日数据”区，不覆盖整个主内容区；在暗色模式与壳层先出现的情况下，用户仍可能感知为大面积空白。
- 只读排查与用户补充信息表明：问题主要发生在“登录后首进”，且是“壳层先出现，主内容很久不出来”，本地 `npm run dev` 环境可复现。

## Proposed Changes
### 1. 优化登录后首跳预热
- 文件：`src/store/modules/auth.ts`、`src/router/route-performance.ts`
- What：在登录成功后、正式 `router.replace` 前，显式预热 `appLayout` 与默认首跳 `dashboard`，必要时同时预热回跳目标命中的首个业务页。
- Why：当前预热在 `afterEach` 才发生，无法帮助“第一次进入主系统”的首跳体验。
- How：复用现有 `routeViewLoaders` / 统一预热能力，不新增第二套路由映射；仅补登录后首跳预热入口。

### 2. 扩大主内容区的可见兜底占位
- 文件：`src/views/dashboard/DashboardView.vue`
- What：把当前仅覆盖统计卡片的 loading 升级为“页面级骨架布局”，至少覆盖欢迎区、统计区、快捷入口区的首屏框架。
- Why：用户当前已经能看到壳层，但主内容区长时间无反馈；即使接口慢，也应立即看到页面骨架，而不是黑色空白区。
- How：保留现有数据结构与请求逻辑，只调整模板的 loading 呈现方式；避免影响权限派生、快捷入口跳转和现有卡片结构。

### 3. 校正壳层与子页面的切换反馈节奏
- 文件：`src/layout/AppLayout.vue`
- What：检查并调整 `Suspense fallback`、`transition` 与 `KeepAlive` 的组合，让业务页异步加载时始终有稳定可见的 fallback；必要时让 fallback 在暗色模式下提升对比度。
- Why：当前用户能看到壳层但看不到主内容，说明壳层内的异步反馈仍不够稳定或不够显眼。
- How：保持缓存策略 `resolveViewKey()` 不变，不改路由权限逻辑，只优化 fallback 的覆盖时机和可见度。

### 4. 为该问题补充可回归验证
- 文件：`scripts/verify-enterprise-core-paths.mjs`、必要时 `src/router/index.ts` / `src/store/modules/auth.ts` 的静态断言位置
- What：把“登录后首进主系统必须具备首跳预热 + 主内容可见占位”加入现有性能回归验证。
- Why：该问题属于性能与感知加载体验问题，修完后必须纳入强制回归，符合当前项目规则。
- How：优先增加静态断言与已有验证入口复用；若适合，再补充对关键首跳路径的可追溯检查项。

## Assumptions & Decisions
- 目标先聚焦“登录后首进系统慢显示”，不扩展到线上部署、MySQL 慢查询或刷新后 `/auth/me` 链路，除非实现中发现二者强耦合。
- 本次优先做“感知性能修复 + 首跳预热优化”，不先做后端接口重构，因为当前现象更接近前端首屏反馈不足。
- 保留现有性能治理基线：懒加载、`keepAlive`、`preloadTargets`、`useStableRequest`、`verify:performance` 不回退。
- 不改变现有登录成功后的业务跳转规则，只缩短等待并增强反馈。

## Verification Steps
1. 运行前端构建：`npm run build`。
2. 运行后端类型检查：`cd backend && npm run check`。
3. 运行性能验证入口：`npm run verify:performance`，确保首跳预热与核心路径验证不回退。
4. 本地手动验证：`npm run dev` 后登录，确认“侧边栏已出现时，主内容区立即显示页面骨架或真实内容”，不再出现长时间黑屏空白。
5. 暗色模式复测：确认工作台 loading 态在暗色模式下仍有足够可见度。
