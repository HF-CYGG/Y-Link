# 全页面铺满右侧屏幕与多端表格布局优化计划

## 一、Summary

本次优化目标是让后台所有业务页面更充分地利用右侧主内容区，减少当前桌面端页面整体偏窄、表格内容显示不全、左右空白浪费严重的问题；同时同步重构平板与手机端的内容排布，让筛选区、卡片区、分页区在不同设备上都保持“占满可用空间但不过度拥挤”的观感。

成功标准：
- 桌面端页面主体默认铺满右侧主区域，不再被过小的最大宽度限制。
- 用户管理、审计日志等表格页减少横向拖动频率，关键信息优先完整展示。
- 平板端提升宽度利用率，避免内容收得过窄。
- 手机端保持单列与安全留白，但顶部工具栏、信息卡、分页区更顺畅地占满屏宽。
- 通用容器改完后，其它业务页面能自动继承收益，不需要每个页面完全重写。

## 二、Current State Analysis

### 1. 右侧主区域存在通用限宽

- 文件：[PageContainer.vue](file:///f:/Y-Link/src/components/common/page-container/PageContainer.vue)
- 当前桌面端默认 `max-w-[1360px]`，平板 `max-w-[1080px]`，并且整页使用 `mx-auto` 居中。
- 这会导致右侧主区域即使还有大量空间，页面主体也不会继续展开。

### 2. 外层主布局仍有较保守留白

- 文件：[AppLayout.vue](file:///f:/Y-Link/src/layout/AppLayout.vue)
- 当前主区域 desktop 为 `px-6 py-5`，tablet 为 `px-4 py-4`，phone 为 `px-3 py-3`。
- 该留白本身不是唯一问题，但与 PageContainer 最大宽度叠加后，会进一步放大“页面偏窄”的感知。

### 3. 表格公共壳层继续收紧横向展示

- 文件：[BizResponsiveDataCollectionShell.vue](file:///f:/Y-Link/src/components/common/business-composite/BizResponsiveDataCollectionShell.vue)
- 默认 `overflow-hidden`、表格容器自带 `p-4`，适合控溢出，但在宽表场景下会让可视宽度更紧。

### 4. 用户管理与审计日志页是当前最明显的受影响页面

- 文件：[UserManageView.vue](file:///f:/Y-Link/src/views/system/UserManageView.vue)
- 文件：[AuditLogView.vue](file:///f:/Y-Link/src/views/system/AuditLogView.vue)
- 两页都走 `PageContainer + PageToolbarCard + BizResponsiveDataCollectionShell + el-table` 结构。
- 表格列本身最小宽度较多，导致在当前容器宽度下，内容显示不全、用户需要频繁左右拖动。

### 5. 多端布局策略已有基础，但还未针对“充分铺满空间”做统一升级

- 平板端当前仍偏保守，很多内容区并没有充分利用 768~1199 的宽度。
- 手机端虽已切换卡片布局，但部分工具栏、提示区、分页区仍可进一步优化铺满感与层次感。

## 三、Proposed Changes

### 1. 调整全局页面容器宽度策略

- 文件：[PageContainer.vue](file:///f:/Y-Link/src/components/common/page-container/PageContainer.vue)
- 变更内容：
  - 取消当前过小的桌面/平板 `max-width` 策略，改为“默认铺满右侧主区域”的容器逻辑。
  - 保留 `narrow` 能力，但改为仅服务少数需要阅读聚焦的页面。
  - 优化各端 section padding，避免容器放开后内容直接贴边或失去层次。
- 目的：
  - 让所有使用 `PageContainer` 的业务页自动获得更宽的展示空间。
  - 把“铺满右侧屏幕”做成通用能力，而不是逐页临时放宽。

### 2. 收紧 AppLayout 的外层留白策略，但保留美观安全边距

- 文件：[AppLayout.vue](file:///f:/Y-Link/src/layout/AppLayout.vue)
- 变更内容：
  - 下调 desktop / tablet / phone 的主内容外层 padding。
  - 保证页面在宽度释放后仍保有统一边距，不出现“内容紧贴边缘”的粗糙感。
- 目的：
  - 配合 PageContainer 扩宽，进一步减少右侧空白浪费。
  - 让屏幕利用率提升，但不破坏当前企业后台的秩序感。

### 3. 升级数据列表壳层的表格容器布局

- 文件：[BizResponsiveDataCollectionShell.vue](file:///f:/Y-Link/src/components/common/business-composite/BizResponsiveDataCollectionShell.vue)
- 变更内容：
  - 优化桌面表格包装容器的 padding 与溢出策略。
  - 保留安全裁剪，但给表格本体更大的实际可用宽度。
  - 评估是否需要把桌面表格模式与平板卡片模式之间增加更细粒度控制。
- 目的：
  - 降低宽表页面被额外包装层“二次收窄”的问题。
  - 让用户管理、审计日志等共用组件直接受益。

### 4. 重构用户管理页的桌面 / 平板 / 手机排布

- 文件：[UserManageView.vue](file:///f:/Y-Link/src/views/system/UserManageView.vue)
- 变更内容：
  - 放宽页面信息区、工具栏、列表区的整体宽度利用率。
  - 调整表格列宽分配，让“账号 / 姓名 / 角色 / 状态 / 权限 / 时间 / 操作”更符合宽屏展示。
  - 在桌面端优先让关键列可见，减少不必要横向滚动。
  - 在平板端优化工具栏换行与信息卡布局。
  - 在手机端保持卡片模式，但优化按钮、说明区、分页条铺满效果。
- 目的：
  - 直接解决截图中“页面中央一小块内容、两边大量空白”的问题。
  - 提升治理页的信息密度与操作效率。

### 5. 重构审计日志页的桌面 / 平板 / 手机排布

- 文件：[AuditLogView.vue](file:///f:/Y-Link/src/views/system/AuditLogView.vue)
- 变更内容：
  - 释放审计日志列表的横向可用空间。
  - 优化筛选栏输入项、时间范围、导出按钮的排布方式。
  - 调整表格列宽与详情列的展示策略，让详情不再过早被压缩。
  - 在平板与手机端同步优化卡片信息层级与留白。
- 目的：
  - 解决审计日志页内容偏窄、详情列阅读体验差的问题。
  - 保持多条件筛选区与数据区的整体平衡感。

### 6. 复核共用分页与工具栏组件的多端适配

- 文件：[PageToolbarCard.vue](file:///f:/Y-Link/src/components/common/page-shared/PageToolbarCard.vue)
- 文件：[PagePaginationBar.vue](file:///f:/Y-Link/src/components/common/page-shared/PagePaginationBar.vue)（需实现前再次复核当前结构）
- 变更内容：
  - 检查工具栏动作区与分页区在更宽布局下是否仍然居中偏窄或换行不佳。
  - 若存在明显空间浪费或布局失衡，做轻量联调而非大改。
- 目的：
  - 保证“页面铺满”后，顶部工具栏与底部分页不会出现新的视觉断层。

## 四、Assumptions & Decisions

### 已确认决策

- 本次范围不是只修“用户管理”和“审计日志”两个页面，而是**以通用容器为主、重点页面为验证样板**，让所有页面都更充分利用右侧屏幕空间。
- 桌面端要明确从“中间收窄的内容岛”改为“铺满右侧主区域”的布局方向。
- 平板与手机端需要同步优化，不接受只修桌面端。
- 目标是“铺满 + 美观”，不是单纯去掉所有 padding 或把内容硬拉满。

### 规划内假设

- 当前多数业务页都复用了 `PageContainer`，因此修改该组件会形成全局收益。
- 少数确实需要更窄阅读宽度的页面，后续仍可通过 `narrow` 保持特殊布局，不影响本次通用放宽策略。
- 当前截图暴露的主要痛点集中在系统治理页，因此用户管理与审计日志将作为首批重点回归对象。

### 暂不纳入本次范围

- 不改动左侧导航结构与宽度策略。
- 不重做页面信息架构，不新增大功能模块。
- 不在本次中处理与布局无关的表格业务逻辑问题。

## 五、Verification Steps

### 桌面端验证

- 打开用户管理页，确认主体占满右侧主区域，左右空白明显减少。
- 检查表格列可见性是否提升，横向拖动频率是否明显下降。
- 打开审计日志页，确认筛选区、提示区、表格区宽度分配更均衡，详情列阅读性提升。

### 平板端验证

- 切换到平板断点，确认工具栏可自然换行，卡片/列表区充分利用可用宽度。
- 检查筛选控件与操作按钮不拥挤、不留大块空白。

### 手机端验证

- 确认页面保持单列阅读，卡片、按钮、分页区都能自然铺满屏宽。
- 检查说明条、工具栏、弹窗相关内容没有因容器放宽而出现贴边或错位。

### 回归验证

- 至少验证用户管理、审计日志、一个非系统治理页面（如工作台或出库列表）是否同步受益。
- 运行前端构建，确保类型与打包通过。
- 若涉及核心布局结构调整，按项目规范执行性能与回归验证。
