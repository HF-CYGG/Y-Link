# 导出 PDF 功能落地计划

## 一、Summary

- 目标：将“导出PDF（预留）”落地为可用功能，且导出的 PDF 与当前打印凭证模板在结构与视觉上保持一致。
- 原则：轻量化、易维护、模块化。
  - 轻量化：导出能力按需加载，不影响常规列表/详情首屏性能。
  - 易维护：导出逻辑与页面逻辑解耦，页面只负责触发与提示。
  - 模块化：抽离独立 PDF 导出模块，可复用于后续单据导出。

## 二、Current State Analysis

- 现有页面已具备：
  - 凭证确认弹窗、打印入口、隐藏打印根节点；
  - “导出PDF（预留）”按钮与 `handleExportVoucherPdf` 占位逻辑；
  - 凭证模板已独立组件化。
- 当前导出问题：
  - 依赖 `globalThis.html2pdf`，但项目未安装/注入 html2pdf 依赖，功能不可用。
  - 导出逻辑直接写在页面内，模块边界不清晰，后续扩展难。
  - 需进一步保证“导出PDF ≈ 打印模板”一致性（同一 DOM、同一样式、同一版芯）。
- 相关文件：
  - [OrderListView.vue](file:///f:/Y-Link/src/views/order-list/OrderListView.vue)
  - [OrderVoucherTemplate.vue](file:///f:/Y-Link/src/views/order-list/components/OrderVoucherTemplate.vue)
  - [package.json](file:///f:/Y-Link/package.json)
  - [vite-env.d.ts](file:///f:/Y-Link/src/vite-env.d.ts)

## 三、Proposed Changes

### 1) 引入并按需加载导出依赖（轻量化）

- 文件：
  - [package.json](file:///f:/Y-Link/package.json)
- 变更：
  - 新增 `html2pdf.js` 依赖（内部包含 html2canvas + jsPDF 组合能力）。
  - 保持“动态 import”方式调用，不在常规路径提前打包执行。
- 原因：
  - 相比页面内全局注入脚本，更可控、更易维护；
  - 动态加载可避免对非导出场景增加运行负担。

### 2) 抽离导出模块（模块化）

- 新增文件：
  - `src/utils/pdf/export-voucher-pdf.ts`
  - `src/types/html2pdf.d.ts`（或等价声明文件）
- 变更：
  - 在 `export-voucher-pdf.ts` 中封装统一能力：
    - 接收导出目标元素、文件名、版芯参数；
    - 统一配置 `margin/image/html2canvas/jsPDF/pagebreak`；
    - 统一错误抛出与可读错误信息。
  - 页面层仅保留“调用 + UI提示”。
- 原因：
  - 导出配置集中管理，后续票据模板复用成本低；
  - 避免页面文件继续膨胀。

### 3) 页面接线与开关策略（易维护）

- 文件：
  - [OrderListView.vue](file:///f:/Y-Link/src/views/order-list/OrderListView.vue)
- 变更：
  - 将当前 `globalThis.html2pdf` 调用替换为模块调用；
  - 保留 `VITE_ORDER_VOUCHER_HTML2PDF_ENABLED` 特性开关；
  - 导出前统一校验：
    - 当前订单存在；
    - 导出根节点存在；
    - 模块加载成功；
  - 失败时统一弹出明确错误提示。
- 原因：
  - 不破坏现有功能路径；
  - 降低生产环境误开关风险。

### 4) “导出与打印一致”策略（核心）

- 文件：
  - [OrderVoucherTemplate.vue](file:///f:/Y-Link/src/views/order-list/components/OrderVoucherTemplate.vue)
  - [OrderListView.vue](file:///f:/Y-Link/src/views/order-list/OrderListView.vue)
- 变更：
  - 导出与打印都基于同一 `.voucher-sheet` DOM（隐藏打印根节点）；
  - 固定 A4 版芯参数（portrait、mm、同边距策略）；
  - 启用 CSS 分页断点策略，保证多商品跨页时表头与行不乱；
  - 对导出路径补充与打印一致的版芯宽度约束。
- 原因：
  - 不做“双模板”，避免样式分叉；
  - 从源头保证一致性。

### 5) 质量保障与回归

- 文件：
  - [OrderListView.vue](file:///f:/Y-Link/src/views/order-list/OrderListView.vue)
  - [OrderVoucherTemplate.vue](file:///f:/Y-Link/src/views/order-list/components/OrderVoucherTemplate.vue)
  - 新增导出模块文件
- 变更：
  - 完成 TypeScript 诊断清零（至少涉及改动文件）；
  - 重点回归：
    - 单商品导出；
    - 多商品跨页导出；
    - 备注超长导出；
    - 打印与PDF视觉一致性核对。

## 四、Assumptions & Decisions

- 决策 1：采用前端导出方案（html2pdf.js 动态加载），不引入服务端渲染 PDF。
- 决策 2：保持“同一模板同一DOM”策略，导出与打印共同复用 `OrderVoucherTemplate`。
- 决策 3：保留环境开关 `VITE_ORDER_VOUCHER_HTML2PDF_ENABLED`，默认可控启用。
- 决策 4：导出结果以“版式一致”为优先，允许浏览器 PDF 引擎导致的极轻微字距差异，但不允许结构差异（缺行、错页、断表头）。
- 决策 5：不新增重型依赖（如浏览器内核级渲染服务），维持前端轻量实现。

## 五、Verification Steps

1. 代码层验证
   - 改动文件 TypeScript/IDE 诊断为 0 错误。
2. 功能路径验证
   - 打开“确认出库信息”弹窗，点击“导出PDF”可成功下载文件。
3. 一致性验证（打印 vs 导出）
   - 对比标题、信息表、明细表、汇总行、签字区位置与样式；
   - 对比单页场景和多页场景结构一致性。
4. 边界验证
   - 订单无明细/备注为空/备注超长/商品行数较多；
   - 导出失败时用户提示准确且不影响后续操作。
5. 性能验证
   - 不触发导出时，页面功能与加载性能保持原状（导出依赖不提前执行）。
